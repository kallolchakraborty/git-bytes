Git Best Practices

schedule2 min readsignal_cellular_1_barFoundational
careerbest-practicesgit

Master Git workflows with proven conventions for commit hygiene, branching strategies, and team collaboration patterns used by top engineering organizations.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • git add -p — interactively stage hunks for atomic commits.
  • git commit -m "type(scope): description" — conventional commits format.
  • git rebase -i HEAD~N — squash, fixup, reorder commits before pushing.
  • git log --oneline --graph --decorate — visualize branch topology.

Staff+ Patterns

  • Atomic commits: one logical change per commit. Makes git bisect precise and reversion clean.
  • Trunk-based development: short-lived branches (<48h), feature flags for incomplete work, daily merges to main.
  • PRs under 400 lines — easier to review, faster to merge, less conflict surface.

Watch Outs

  • Merging main into a feature branch daily creates ugly merge bubbles. Rebase instead.
  • Large commits touching 50+ files (big bang) — impossible to review, risky to revert. Use git add -p.
  • Committing config files (.env, IDE settings) — always verify with git status and .gitignore.

Interview Angle

Amazon L6: "How do you enforce commit message conventions across a 500-engineer org?" Answer: commitlint in CI + pre-receive hook. The hook rejects non-compliant commits before they reach the remote. Combine with gh api to audit compliance weekly.

Commit Conventions

Conventional Commits provide a standardized format: type(scope): description. Types include feat, fix, chore, docs, refactor, test, perf. This enables automated changelog generation, semantic versioning, and readable history. Each commit should be atomic — one logical change per commit. Use git add -p to stage hunks selectively and craft clean commits.

language-bash
# Conventional commit format
git commit -m "feat(auth): add OAuth2 login flow"
git commit -m "fix(api): handle null pointer in user serializer"
git commit -m "refactor(db): extract connection pool into helper"

# Write descriptive bodies for complex changes
git commit -m "perf(cache): reduce TTL on session keys\n\nBenchmarking showed stale sessions caused 15% cache miss rate.\nReducing TTL from 3600s to 300s improves hit ratio."

Branching Strategies

GitFlow: main, develop, feature/*, release/*, hotfix/* — best for scheduled releases with multiple supported versions. GitHub Flow: single main branch with short-lived feature branches merged via PR — ideal for continuous deployment. Trunk-Based: all developers merge to main daily with short-lived branches (<48h) — requires feature flags and robust CI.

language-bash
# GitFlow branches
git checkout -b feature/user-auth develop
git checkout -b release/v2.1.0 develop
git checkout -b hotfix/security-patch main

# GitHub Flow
git checkout -b feat/login-form main
# ... work, push, open PR ...
git checkout main && git merge --squash feat/login-form

# Trunk-Based (short-lived branch)
git checkout -b ff-user-auth main
# merge within 24-48 hours
git checkout main && git merge ff-user-auth

Workflow Patterns

Forking Workflow: external contributors fork the repo, push changes, and submit cross-repo PRs. Feature Branch Workflow: all changes developed in dedicated branches off main. Gitflow Workflow: elaborate branching for release management. Maintenance Branch Workflow: separate branches for LTS releases with backport policies.

language-bash
# Forking workflow (external contributor)
git remote add upstream https://github.com/org/repo.git
git fetch upstream
git checkout -b feature/contribution upstream/main
# ... make changes ...
git push origin feature/contribution
# Then open PR from fork to upstream

# Feature branch workflow (internal)
git checkout -b feat/payment-gateway main
git rebase main  # keep up to date
git push origin feat/payment-gateway

Team Guidelines

Establish a team Git contract: branch naming (feat/, fix/, chore/, docs/), commit message format, PR size limits (< 400 lines), squash vs merge commit policy, and required status checks. Enforce with branch protection rules: require PR reviews, status checks, up-to-date branches, and signed commits.

language-bash
# Branch naming conventions
git checkout -b feat/issue-123-stripe-integration
git checkout -b fix/regression-login-redirect
git checkout -b chore/upgrade-deps-2024

# Commit message template (#.gitmessage)
# 50-character subject line
#
# 72-character wrapped body explaining what and why
# Reference issues: Closes #123, Ref #456
#
# Co-authored-by: Partner <email>

Automated Checks

Client-side hooks via .git/hooks/pre-commit run linters, formatters, and secret scanners before each commit. Server-side via GitHub Actions enforce CI/CD: build, test, lint, type check, and security scan. Use commitlint and husky for commit message validation. git-secrets or truffleHog prevent credential leaks.

language-bash
# .husky/pre-commit
npx lint-staged
npm run typecheck

# .github/workflows/ci.yml (server-side)
name: CI
on: [pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run lint
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test && npm run test:e2e

# Client-side secret scanner
git secrets --scan
lightbulbDeep Dive & Key Takeawaysexpand_more

Git best practices are not one-size-fits-all. Small teams (2-5) benefit from GitHub Flow's simplicity. Growing teams (10-50) need branch protection rules and conventional commits. Large orgs (50+) require trunk-based development with feature flags and automated release engineering. The key metrics to track: time from commit to deployment, merge conflict frequency, and revert rate. Invest in automation proportional to team size — manual gates that work for 5 people break at 50.
