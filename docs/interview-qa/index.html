Core Commands

  • git cat-file -p HEAD — inspect the commit object (tree, parent, author).
  • git rev-parse HEAD — resolve any ref to a SHA. Plumbing command, signals depth.
  • git merge-base A B — find the common ancestor. Core of the recursive merge algorithm.
  • git bisect run ./test.sh — automate regression hunting.
  • git reflog — safety net. Recover lost commits.

Staff+ Patterns

  • Know the three Git data structures: objects (content-addressable), refs (pointers), and the index (manifest).
  • For system design: partial clone (--filter=tree:0), sparse checkout (cone mode), Scalar, and VFS for Git.
  • Design a merge queue: batch compatible PRs, squash-merge to integration branch, run CI, push sequentially.

Watch Outs

  • Confusing reset --soft vs --mixed vs --hard — common interview mistake.
  • Not knowing what happens during git push (pack-objects → send-pack → receive-pack → update-ref).
  • Forgetting git bisect — it's the #1 debugging tool in Git interviews.

Interview Angle

Know company-specific workflows: Google (Piper/trunk-based), Meta (Mercurial/Mononoke), Microsoft (Scalar/VFS). For L7+, design a version control system for 100K engineers — discuss sharding, virtual filesystems, and custom protocol negotiation.

Fundamental Questions (L3–L4)

Cover the basics: What is a commit? A snapshot (tree object) with parent pointers. Difference between git merge and git rebase? Merge preserves history, rebase rewrites it. Three Git objects: blob (file), tree (directory), commit (snapshot). HEAD is a symbolic ref to the current branch tip. Index (staging area) is a manifest of what will go into the next commit.

language-bash
# Q: How does git store a commit?
# A: As an object with tree, parent, author, committer, message

# Inspect objects
git cat-file -p HEAD      # view commit object
git cat-file -p HEAD^{tree} # view tree object
git ls-tree HEAD          # list tree entries

# Q: Difference between git pull and git fetch?
git fetch origin          # download objects + refs, don't merge
git pull origin main      # fetch + merge (or rebase with --rebase)

# Q: How to undo a pushed commit?
git revert HEAD           # creates inverse commit (safe)
git reset --hard HEAD~1   # local only; force push if needed

Intermediate Questions (L5 — Senior)

Topics: git stash internals (temporary commits), reflog as safety net (logs HEAD movements), cherry-pick commits across branches, bisect for binary search debugging, submodules vs subtrees, worktree for parallel development. Know the difference between FETCH_HEAD, ORIG_HEAD, and MERGE_HEAD.

language-bash
# Q: How to recover a lost commit?
git reflog                 # show HEAD history
git checkout HEAD@{2}      # checkout a reflog entry
git cherry-pick <sha>      # re-apply lost commit

# Q: How to debug a regression?
git bisect start
git bisect bad HEAD
git bisect good v1.0
# git bisect checks out midpoint — test and mark
git bisect good          # or git bisect bad
git bisect reset         # end bisect session

# Q: Merge vs Rebase?
git merge feature         # creates merge commit
git rebase feature        # replays commits on top
# (no merge commit, linear history)

Advanced Git Questions (L5–L6)

Deep topics: git gc and object expiration, packfile format (thin vs thick), --filter=blob:none for partial clone, git replace for object replacement, git notes for metadata, git fsck for repository integrity, git archive for export, and git range-diff for comparing patch series.

language-bash
# Q: How does git garbage collection work?
git gc --auto             # runs when loose objects exceed limit
git count-objects -v      # view object store stats

# Q: How to reduce clone size?
git clone --filter=blob:none <url>  # omit blobs, fetch on demand
git clone --depth=1 <url>           # shallow clone (no history)

# Q: How to compare two versions of a branch?
git range-diff main~2..main feature~2..feature
# Shows how patch series changed between rebases

# Q: Repository integrity check?
git fsck --full           # check all objects
# git fsck --lost-found   # find dangling commits/blobs

Staff+ System Design (L6–L8)

System design questions: Design a monorepo system handling 10K developers with partial clone + sparse checkout + VFS for Git. Design merge queue architecture with batching and semantic conflict detection. Design a CI pipeline that builds only affected projects (dependency graph). Design a migration from SVN to Git with full history preservation. Design custom merge drivers for DSL config files.

language-bash
# System design: Monorepo for 10K developers
# 1. Partial clone: --filter=tree:0 (omit all blobs)
# 2. Sparse checkout: cone mode for project-scoped files
# 3. Scalar: Microsoft's enhanced Git client
# 4. Build system: Bazel with remote caching

# System design: Merge Queue
# 1. PR author rebases on main
# 2. Queue picks batch of compatible PRs
# 3. Squash-merge to temporary integration branch
# 4. Run full CI on integration branch
# 5. If green, push commits sequentially to main
# 6. If red, drop offending PR, retry remaining

Google-Specific Questions

Piper (Google's monorepo) is not Git — but Google asks Git fundamentals for context. Google L4–L5: "How would you design git log --graph?" "How does git push work end-to-end?" Google L6+: "Design a protocol for Git that supports 100K developers." "How would you add partial clone support to a custom VCS?" Google also asks about trunk-based development at scale — how to keep main green with 10K engineers committing daily. Common question: "Explain how you'd implement git bisect on a server-side monorepo without checking out every commit."

language-bash
# Google L6: Design a merge-on-server protocol
# 1. Client sends OID + want/have lines
# 2. Server computes merge-base via commit-graph
# 3. Server generates pack for missing objects
# 4. Client receives thin pack, resolves deltas

# Google L5: How git push works end-to-end
# 1. Client: index-pack, pack-objects, send-pack
# 2. Server: receive-pack, unpack-objects, update-ref
# 3. Hooks: pre-receive, update, post-receive
# 4. Edge: push options, report-status, atomic push

Meta (Facebook) Questions

Meta uses Mercurial (Mononoke) — interviewers look for transferable Git depth. L5: "What happens when you type git commit?" Walk through hash computation, tree creation, ref update. L6: "Design a Git-to-Mercurial bridge." "How would you handle 10K commits/min in a monorepo?" L7: "Design a virtual filesystem for Git that only materializes files on access." "How would you implement blame annotation at scale using bitmap indexes?" Meta values performance optimization — know git fsmonitor (File System Monitor), git sparse-checkout cone mode, and the untracked cache.

language-bash
# Meta L6: Virtual filesystem for Git
# 1. Placeholder files in working tree (reparse points)
# 2. On read: fetch blob from remote, populate, cache
# 3. On write: convert placeholder to full file
# 4. Use filesystem filter driver (e.g., Git LFS pattern)
# 5. On status: only scan materialized directories

# Meta L5: Anatomy of git commit
# 1. Index diff → tree object (hash of sorted entries)
# 2. Commit object: tree_sha + parent_sha + author + committer + message
# 3. Write to .git/objects/XX/XXXXXXXXXXXXXXXXX
# 4. Update HEAD: refs/heads/<branch> → new SHA
# 5. Reflog entry: HEAD@{0}: commit: <message>

Amazon Questions

Amazon uses Git internally and asks about BR (Branches) and CR (Code Review) workflows. L5: "How would you automate a git bisect to find which commit broke a deployment?" "What's the difference between reset, checkout, and revert?" L6 (SDE III): "Design a CI/CD pipeline where each commit triggers tests for only affected services — how do you determine the dependency graph from Git history?" L7 (Principal): "Design a distributed version control system for Amazon's internal monorepo with 100K engineers." Leadership Principles tested: Bias for Action (force-push during SEV?), Dive Deep (explain packfile bitmaps), Disagree and Commit (merge strategy disagreement).

language-bash
# Amazon L6: Dependency-aware CI from git diff
git diff --name-only HEAD~1 HEAD  # changed files
# Parse BUILD files to find affected targets
# for each changed file:
#   bazel query "rdeps(//..., //path/to/changed:target)"
# Run tests for affected + dependent targets only

# Amazon L5: Bisect automation
git bisect start HEAD <known-good>
git bisect run ./test_deployment.sh
# Exits 0 = good, non-0 = bad
# git finds the exact commit that broke deployment
# Script should be idempotent and fast

Apple Questions

Apple uses Git extensively. L4–L5: "Explain why git merge can create conflicts and git rebase can too — which is safer?" "How would you split a large commit into smaller ones?" L6: "Design a Git workflow for a project with 500 engineers working on the same repo" — Apple emphasizes structured branching: main → release/X.Y → hotfix-*. L7: "How would you scale Git for Xcode's project format (pbxproj) that causes merge conflicts?" "Design a custom merge driver for Xcode project files." Apple also asks about git worktree for parallel builds and git submodule vs git subtree for dependency management.

language-bash
# Apple L7: Custom merge driver for Xcode .pbxproj
# .gitattributes
*.pbxproj merge=xcode-project

# .git/config
[merge "xcode-project"]
  name = Xcode project file merge driver
  driver = xcode-merge %O %A %B %L %P

# The driver must sort UUIDs, merge file references,
# and produce a valid plist. No standard tool exists.

# Apple L6: Structured branching for 500 engineers
# main (stable, release candidates)
#   └── release/3.0 (code freeze branch)
#         ├── hotfix/3.0.1 (from release)
#   └── feature/* (short-lived, squash-merge)
#   └── team/* (long-lived, rebase weekly)

Netflix Questions

Netflix uses Git with trunk-based development and CD. L5: "How does git revert work under the hood?" "What's the difference between --no-ff and --ff-only?" L6: "Design a canary deployment pipeline triggered by Git tags — how do you ensure the Git tag is cryptographically signed before deploying?" "How would you automate detection of bad merges using git range-diff?" L7: "Design a system that runs chaos engineering experiments on Git operations (e.g., network partitions during push)." Netflix values automation — scripting with git rev-list, git for-each-ref, and git merge-base --is-ancestor for tooling.

language-bash
# Netflix L6: Signed tag deployment pipeline
# 1. Developer creates annotated + signed tag
git tag -s v2.1.0 -m "Release v2.1.0"
# 2. CI verifies signature
git tag -v v2.1.0
# 3. Only deploy if signature is from trusted keyring
# 4. Deploy to canary, monitor metrics, roll forward

# Netflix L7: Git chaos engineering
# Test: network partition during force-push
#   → client hangs, retries, may push partial refs
#   → verify receive.fsckObjects catches corruption
# Test: concurrent pushes to same branch
#   → loser gets rejected, must fetch + rebase
# Test: disk full during gc
#   → verify .git keeps consistent state

Microsoft Questions

Microsoft runs the largest Git repository on the planet (Windows: 300GB+). L5: "How does git sparse-checkout work?" "What's the difference between --filter=blob:none and --filter=tree:0?" L6: "Design Scalar — the Git client for large repos — what features would you include?" "How would you reduce git status time from 30s to under 1s?" L7: "Design a Git Virtual File System (VFS) that supports 300GB repos without cloning everything." "How would you implement git fetch over a high-latency satellite link?" Microsoft values Scalar, GVFS (VFS for Git), partial clone, and prefetch — understand these deeply.

language-bash
# Microsoft L6: Scalar design for large repos
# Features Scalar provides:
# 1. Partial clone: git clone --filter=tree:0 --no-checkout
# 2. Sparse checkout: git sparse-checkout init --cone
# 3. Background maintenance: git maintenance start
# 4. Filesystem monitor: git fsmonitor--daemon start
# 5. Prefetch: periodic background fetch of known refs
# 6. Multi-pack index (MIDX): fast object lookups

# How git status becomes fast:
# Without fsmonitor: lstat() every file → O(n) syscalls
# With fsmonitor: kernel notifies on file changes → O(1)
# Integration: watchman-or-fsmonitor hook → update-index

Stripe / Uber / Snowflake Questions

Stripe (L5): "How would you use git notes to attach metadata to a commit without changing its SHA?" "Design a workflow where every commit message must include an issue ID — enforced by server-side hook." Uber (L5–L6): "How do you handle Git in a microservices monorepo with 2000 services?" "Design an automated system to detect unused files in a monorepo using git log --diff-filter." Snowflake (L6): "How would you migrate a repository from Perforce to Git while preserving all history and commit metadata?" "Design a GitOps workflow for deploying infrastructure changes — how do you handle rollback of a bad deployment?"

language-bash
# Stripe L5: Enforce issue IDs in commit messages
# .git/hooks/commit-msg
#!/bin/sh
ISSUE_REGEX="^(feat|fix|chore)\([^)]+\): [A-Z]+-[0-9]+ .+"
if ! grep -qE "$ISSUE_REGEX" "$1"; then
  echo "ERROR: Commit must include issue ID (e.g., STR-123)"
  exit 1
fi

# Uber L6: Detect unused files in monorepo
# Files that haven't been modified in 2 years:
git log --diff-filter=A --name-only --since="2 years ago" |
  sort -u | while read f; do
  if ! git log --oneline -- "$f" | grep -q .; then
    echo "Unmodified for 2+ years: $f"
  fi
done

lightbulbDeep Dive & Key Takeawaysexpand_more

Git interviews at FAANG+ companies test three levels: L4/L5 (Mid/Senior): fundamental commands, merge vs rebase, undoing changes. L6 (Staff): git internals, object model, performance tuning, debugging complex scenarios. L7+ (Principal/Distinguished): system design for version control at scale, monorepo architectures, custom tooling. Common gotchas: candidates confuse reset --soft vs --mixed vs --hard; can't explain what happens during git push (pack-objects, ref update, receive-pack); don't know how to recover from a force-push mistake. Practice with git bisect run scripting and git filter-repo for history rewriting questions.

Top tips from FAANG interviewers: (1) Always draw the commit graph — visualize what happens during merge, rebase, cherry-pick. (2) Know the three Git data structures: objects (content-addressable), refs (pointers), and the index (manifest). (3) For Staff+, always discuss trade-offs: performance vs consistency, speed vs safety. (4) Use git help -a to discover plumbing commands — knowing git rev-parse, git hash-object, and git update-ref signals deep understanding. (5) For system design, reference real systems: Google Piper, Meta Mononoke, Microsoft Scalar, GitHub's load-balanced Git architecture.
