Policy as Code

schedule1 min readsignal_cellular_4_barAdvanced / Staff+
staff+enterprisepolicysecurity

Policy as Code codifies governance rules in version-controlled configuration files, evaluated deterministically across your repository hierarchy.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • gh api /repos/{owner}/{repo}/rulesets — list, create, update rulesets via REST.
  • terraform import github_repository_ruleset.main 123 — manage rulesets declaratively.
  • gh api /repos/{owner}/{repo}/rulesets/{id}/evaluate — dry-run a ruleset before enforcement.

Staff+ Patterns

  • Store rulesets in .github/rulesets.yml for version-controlled, auditable policy deployment.
  • Use Terraform github_repository_ruleset resource for org-wide baseline enforcement.
  • Run a nightly Action to audit policy coverage — alert when repos lack required rulesets.

Watch Outs

  • Rulesets in evaluate mode log violations but don't block — missing this means policies aren't actually enforced.
  • Only the first matching ruleset applies per ref target. If global ruleset matches, org ruleset won't run.
  • Custom roles and rulesets are separate concepts — know when to use each.

Interview Angle

Stripe L6: "Design a policy-as-code system for 500 repos that enforces branch protection, commit signing, and secret scanning. How do you handle exceptions?" Key: a central policies repo with opt-out via approved PR to an exceptions file. Every opt-out has an expiry date.

Branch Protection Rulesets

Modern branch protection uses rulesets with required status checks, linear history, force push restrictions, and required reviewers. Rulesets are evaluated in priority order (BY_INDEX). Each rule targets a ref pattern and specifies enforcement status (active, evaluate, disabled). Use evaluate mode to test rules before enforcing.

name: protect-main
enforcement: active
bypass_mode: always
conditions:
  refName:
    include: ["refs/heads/main"]
rules:
  - type: required_status_check
    parameters:
      strict: true
      required_status_checks: ["CI", "lint"]

Repo Creation Policies

Enterprise admins can restrict repo creation visibility to private only, enforce naming conventions via pre-receive hooks, and auto-apply baseline rulesets. Configure repo_creation policies in the enterprise settings or via the GraphQL EnterpriseRepositoryCreationPolicy mutation.

mutation {
  updateEnterpriseRepositoryCreationPolicy(
    enterpriseId: "E_123"
    visibility: PRIVATE_ONLY
    forkedRepoCreation: DISABLED
  ) { policy { id } }
}

Custom Roles

Custom roles combine granular permissions into reusable templates. Instead of the built-in Read/Write/Admin triad, you can create roles like security-auditor with repo:security_events:read and org:dependabot_alerts:read. Assign via gh api /orgs/{org}/organization-roles or the GraphQL createOrganizationRole mutation.

Ruleset Evaluation

The ruleset engine evaluates in a defined order: global enterprise rulesets first, then org, then repo. Only the first matching ruleset applies per target ref. Bypass lists allow specific actors or teams to skip rules. Use the REST API GET /repos/{owner}/{repo}/rulesets to inspect which rulesets apply and the evaluate endpoint to dry-run.

curl -L -X POST https://api.github.com/repos/org/repo/rulesets/1/evaluate
  -H "Accept: application/vnd.github+json"
  -d '{"ref":"refs/heads/feature-branch"}'

Policy Automation

Automate policy deployment via .github/rulesets.yml files, Terraform provider, or the gh CLI. Use Actions to audit policy coverage nightly: check that all repos have required branch protections and report drift. Combine with CODEOWNERS to require platform team approval for policy changes.

name: audit-policies
on:
  schedule: [{ cron: '0 6 * * *' }]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - run: gh api /orgs/$ORG/repos --jq '.[].name' | while read r; do
          gh api repos/$ORG/$r/rulesets || echo "$r missing"
        done
lightbulbDeep Dive & Key Takeawaysexpand_more

Policy as Code shifts governance from checkbox audits to deterministic, version-controlled enforcement. Rulesets replace the older branch protection API with a unified, hierarchical engine. Each ruleset has a conditions block targeting refs, a rules block with enforcement types, and a bypass_mode for exceptions. The evaluation chain starts at the enterprise level — if the first matched ruleset allows the action, evaluation stops. This means you can set a global "require status checks" and an org-level "require two reviewers" and both must be satisfied only if they are in different rulesets targeting different refs. The evaluate endpoint returns which rulesets matched and whether the ref would pass. Use Terraform's github_repository_ruleset resource to manage rulesets declaratively. For pre-receive hooks, GitHub Enterprise Server offers server-side Git hooks for custom validation (e.g., commit message format, file size limits). Combine with SCIM for automated bypass list management — service accounts get different ruleset treatment than human users.
