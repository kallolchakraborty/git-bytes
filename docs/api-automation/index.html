API Automation

schedule1 min readsignal_cellular_4_barAdvanced / Staff+
staff+enterpriseapiautomation

Automating GitHub at scale requires understanding authentication flows, webhook delivery guarantees, and the right API surface for each use case.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • gh api /repos/{owner}/{repo}/check-runs — create check runs with file-level annotations.
  • gh api --paginate /orgs/{org}/repos --jq '.[].name' — list all repos with pagination.
  • gh api /repos/{owner}/{repo}/git/refs — interact with Git refs directly.

Staff+ Patterns

  • Prefer GitHub Apps over OAuth Apps for automation — granular permissions, installation scoping, dedicated rate limits.
  • Use GraphQL for bulk queries — a single repositories query with 100 nodes replaces 100 REST calls.
  • Webhook verification: validate x-hub-signature-256 HMAC before processing payloads.

Watch Outs

  • GitHub App installation tokens expire after 1 hour — regenerate via JWT re-authentication.
  • REST API pagination (page/per_page) is O(n) for deep pages. GraphQL cursor pagination is O(1).
  • Rate limits: REST 5000/hr per user, GitHub App 5000/hr per installation. GraphQL uses point-based costing.

Interview Angle

GitHub L6: "Design a CI bridge using the Checks API." Key: each pipeline step maps to a check run with annotations pointing to the exact file + line of failure. Use POST /repos/{owner}/{repo}/check-runs with output.annotations for rich inline feedback in the PR.

GitHub Apps vs OAuth Apps

GitHub Apps are first-class actors with granular permissions, installation scoping, and rate limit isolation. OAuth Apps act on behalf of a user with a single token. Prefer GitHub Apps for automation: they authenticate via JWT (RS256 signed with app private key), exchange for an installation access token (1h expiry), and scope requests to specific repositories.

# Generate JWT
payload = {
  iat: now - 60, exp: now + 600, iss: app_id
}
token = jwt.encode(payload, private_key, algorithm="RS256")
# Exchange for install token
curl -X POST /app/installations/{id}/access_tokens \
  -H "Authorization: Bearer $jwt"

Checks API

The Checks API lets CI systems create check runs and suites that appear directly in pull requests. Each check run has a name, status (queued/in_progress/completed), conclusion (success/failure/neutral/cancelled), and annotations at the file+line level. Use POST /repos/{owner}/{repo}/check-runs with output.summary and output.annotations for rich feedback.

curl -X POST /repos/org/repo/check-runs \
  -H "Accept: application/vnd.github+json" \
  -d '{"name":"lint","head_sha":"$SHA","status":"completed","conclusion":"success","output":{"title":"ESLint","summary":"0 errors"}}'

Webhook Delivery

Webhooks deliver event payloads to configurable URLs with a retry mechanism. Upon delivery failure (non-2xx), GitHub retries up to 3 times with exponential backoff (~30s, ~2min, ~5min). Each delivery includes a unique x-github-delivery ID and the x-hub-signature-256 HMAC for verification. Set up a dead letter queue for hardened delivery guarantees.

# Verify payload signature
const sig = crypto
  .createHmac('sha256', secret)
  .update(payload)
  .digest('hex')
assert(sig === header)

Batch GraphQL

GraphQL enables batched queries that replace multiple REST calls. Use aliases, fragments, and pagination via first/after cursors. The nodes pattern is efficient for bulk operations. Rate limit is based on node count (5000 points per hour, query cost = cost field in response).

query repos($org: String!) {
  organization(login: $org) {
    repositories(first: 100) {
      nodes { name defaultBranchRef { name } }
      pageInfo { hasNextPage endCursor }
    }
  }
}

Internal Tooling

Build internal developer tooling with GitHub's API surface. Patterns include: gh api in shell scripts, Octokit SDK wrappers, and custom Actions. Use conditional requests with If-None-Match / ETag to reduce bandwidth. Cache API responses with TTLs respecting x-ratelimit-remaining headers.

gh api repos/org/repo --jq '.default_branch'
gh api --paginate /orgs/org/repos --jq '.[].name'

API automation at scale demands careful authentication strategy. GitHub Apps are the gold standard for automation — they authenticate as a bot, scope to installations, and have dedicated rate limits (5000/hour per app, not per user). The JWT flow: generate a JWT signed with your app's private key (RS256, 10 minute max expiry), POST to /app/installations/{id}/access_tokens, and use the returned token for API calls. Tokens expire after 1 hour; re-authenticate by generating a new JWT. For webhook reliability, configure a webhook with a secret, verify HMAC signatures, and respond with 200 within 10 seconds to avoid timeouts. Use the /app/hook/deliveries endpoint to inspect recent deliveries and replay failed ones. GraphQL is preferable for bulk queries — a single repositories query with 100 nodes costs 1 point (plus 1 per node). Always handle pagination with cursors — REST returns Link headers, GraphQL uses pageInfo. For internal tooling, the gh CLI's --paginate flag auto-fetches all pages. Build a CI/CD bridge using the Checks API: each pipeline step maps to a check run with annotations pointing to the exact line of failure.
