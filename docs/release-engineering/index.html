Release Engineering

schedule1 min readsignal_cellular_4_barAdvanced / Staff+
staff+leadershipreleaseengineering

Release engineering automates the process from commit to production deployment with semantic versioning, changelog generation, and hotfix management.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • git tag -s v2.1.0 -m "Release v2.1.0" — create a signed, annotated tag.
  • git cliff -o CHANGELOG.md — auto-generate changelog from conventional commits.
  • npx semantic-release — fully automated version bump, changelog, tag, and GitHub Release.
  • git cherry-pick -x $FIX_COMMIT — hotfix with original commit reference.

Staff+ Patterns

  • Commit message determines version bump deterministically: fix → PATCH, feat → MINOR, BREAKING CHANGE → MAJOR.
  • Use semantic-release or changesets for fully automated release pipelines from conventional commits.
  • Hotfix workflow: branch from tag → cherry-pick → version bump → tag → merge back to main.

Watch Outs

  • Hotfixes that aren't merged back to main cause regressions in the next release — always merge back.
  • Pre-1.0.0: conventional commits treat MINOR as breaking. Set npm version premajor to override.
  • Signed tags (git tag -s) require GPG setup. Without it, releases can't be cryptographically verified.

Interview Angle

Netflix L6: "Design a release pipeline that deploys to canary (1% traffic) and auto-promotes if error rate is below threshold." Key: Git tags as deployment artifacts (deploy-prod-v435), rollback = deploy previous tag, canary duration = 5 min with metric monitoring.

Semantic Versioning

Semantic versioning (MAJOR.MINOR.PATCH) communicates change severity: MAJOR for breaking changes, MINOR for backward-compatible features, PATCH for backward-compatible fixes. Parse commit messages to determine the version bump. Pre-release suffixes (-alpha.1, -beta.2, -rc.1) denote stability. Use semver CLI or libraries for programmatic comparisons.

v2.3.1      # stable release
v2.4.0-alpha.1  # pre-release
v3.0.0-rc.1     # release candidate

Conventional Commits

Conventional Commits standardizes commit messages for automated versioning. Format: type(scope): description. Types include feat (MINOR bump), fix (PATCH bump), BREAKING CHANGE (MAJOR bump), chore, docs, refactor (no bump). Tools like commitlint enforce formatting, and semantic-release automates the pipeline.

feat(cli): add dry-run flag
fix(auth): handle token expiry

BREAKING CHANGE: drop Node 16 support

Changelog Generation

Auto-generate changelogs from conventional commits grouped by type. Order: Breaking Changes, Features, Bug Fixes, Performance, Documentation. Each entry links to the commit and PR. Tools: git-cliff, semantic-release, standard-version. Keep a CHANGELOG.md per the keepachangelog.com format.

## [2.4.0] - 2025-03-15
### Features
- Add dry-run flag (#142)

### Bug Fixes
- Handle token expiry in auth (#138)

Hotfix Management

Hotfixes patch a production release without deploying the latest main. Branch from the release tag, cherry-pick the fix, version bump, and tag. Merge the hotfix back to main to prevent regressions. Use git cherry-pick -x to include the original commit hash. Automate hotfix releases with a dedicated workflow triggered on release/v* branches.

git checkout -b hotfix v2.3.1
git cherry-pick -x $FIX_COMMIT
npm version patch
git push --tags

Release Branching

Release branching strategies vary by maturity: GitHub Flow (release from main with tags), Git Flow (dedicated release branches), or trunk-based with release trains. At scale, use release branches for LTS lines. Each release branch gets backported fixes. Automate branch creation, tag signing, and release note generation with Actions.

name: release
on:
  push:
    branches: [main]
permissions:
  contents: write
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: cycjimmy/semantic-release-action@v4
lightbulbDeep Dive & Key Takeawaysexpand_more

Release engineering is the intersection of version control, CI/CD, and software distribution. The pipeline starts with commit conventions — the commit message determines the version bump through a deterministic parser. semantic-release reads commits since the last tag, determines the next version, updates the changelog, creates a Git tag, and publishes a GitHub Release. The decision flow: fix commits trigger PATCH, feat triggers MINOR, commits with BREAKING CHANGE in the footer trigger MAJOR (pre-1.0.0 treats MINOR as breaking). Pre-releases use channels like beta or next to publish without affecting the main dist-tag. For monorepos, tools like changesets or lerna handle independent versioning per package. Hotfix management is a common source of release engineering failure — always merge hotfixes back to main and verify with merge commits. Release branches should have a short lifespan (merged or closed within days). For compliance, sign Git tags with GPG (git tag -s) and attest releases with Sigstore. Monitor release metrics: time from merge to release, release failure rate, and hotfix frequency as lagging indicators of code quality.
