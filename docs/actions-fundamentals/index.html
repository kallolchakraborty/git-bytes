GitHub Actions Fundamentals

schedule1 min readsignal_cellular_1_barFoundational
githubactionsci/cd

GitHub Actions automates software workflows directly in your repository — from CI/CD to issue management and deployment.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • .github/workflows/ci.yml — workflow file location.
  • on: [push, pull_request] — trigger on events.
  • jobs: test: runs-on: ubuntu-latest — job definition.
  • strategy: matrix: node: [18, 20] — run across versions.

Staff+ Patterns

  • Use workflow_call for reusable workflows — define a deployment workflow once, call it from all repos.
  • Pin actions by commit SHA (actions/checkout@a81bbbf) instead of version tag to prevent supply chain attacks.

Watch Outs

  • GITHUB_TOKEN permissions are permissive by default. Always set permissions: contents: read at job level.
  • Matrix jobs that exceed 256 runners will be queued. Use concurrency limits to control parallel runs.

Interview Angle

Netflix L6 asks about Actions architecture at scale: runner autoscaling, job scheduling, artifact caching. Explain the polling model and how the Actions coordinator manages the global job queue. Know the difference between hosted and self-hosted runners.

Workflow Structure

A workflow is a YAML file in .github/workflows/. Top-level keys: name, on (trigger), jobs. Each job runs on a runner (runs-on) with steps. Steps can use uses (marketplace action) or run (shell command).

name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

Triggers & Events

Workflows trigger on events: push, pull_request, schedule (cron), workflow_dispatch (manual), repository_dispatch (API). Use push: with branches: and paths: filters for precise control. Activity types (e.g., pull_request: [opened, synchronize]) narrow triggers further.

on:
  push:
    branches: [main]
    paths: ['src/**']
  schedule:
    - cron: '0 6 * * 1'

Jobs & Steps

Jobs run in parallel by default; needs: creates sequential dependencies. Steps share data via $GITHUB_ENV and $GITHUB_OUTPUT. Matrix strategies (strategy.matrix) run jobs across OS/version combinations. if: conditional syntax controls step execution.

jobs:
  lint:
    runs-on: ubuntu-latest
  test:
    needs: lint
    strategy:
      matrix:
        node: [18, 20]

Environment Variables

Built-in env vars: $GITHUB_WORKSPACE, $GITHUB_SHA, $GITHUB_REF, $GITHUB_EVENT_PATH (event payload JSON). Secrets stored in Settings > Secrets are accessed via ${{ secrets.MY_SECRET }}. Use environment files ($GITHUB_ENV, $GITHUB_OUTPUT) for inter-step communication.

steps:
  - name: Set env
    run: echo "MY_VAR=hello" >> $GITHUB_ENV
  - name: Use secret
    run: deploy --token ${{ secrets.DEPLOY_TOKEN }}

Marketplace Actions

Pre-built actions from the marketplace: actions/checkout, actions/setup-node, actions/cache, actions/upload-artifact. Pin actions by commit SHA for security. Create custom actions as Docker containers, JavaScript, or composite run steps.

# Popular actions
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
- uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
lightbulbDeep Dive & Key Takeawaysexpand_more

GitHub Actions uses a polling-based runner architecture. The hosted runner pool runs on Azure Standard_DS2_v2 VMs (2 cores, 7 GB RAM) with Ubuntu, Windows Server, or macOS. Each job gets a fresh VM with a clean filesystem. The ACTIONS_RUNNER_DEBUG secret enables step-level debug logging. Billing depends on included minutes (free tier: 2000 min/month for public repos, 500 min for private) — self-hosted runners are free. The GITHUB_TOKEN is auto-generated per job with scoped permissions and expires after the job. For security, use permissions: at the workflow or job level to restrict token access. Artifacts and logs are retained for 90 days by default. Actions support workflow_call (reusable workflows) and composite actions for DRY workflows.
