Branching & Merging

schedule1 min readsignal_cellular_1_barFoundational
gitbranchingmerging

Branching is Git's killer feature — lightweight, cheap, and local. Merging weaves divergent histories back together.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • git branch / git switch -c — create branches. switch is the modern replacement for checkout.
  • git merge --no-ff feature — force a merge commit to preserve branch topology in history.
  • git merge --squash feature — squash all feature commits into one on main.
  • git log --graph --oneline --all — visualize branching topology.

Staff+ Patterns

  • Use --no-ff for feature branches into main — preserves the fact that work happened in parallel. --ff-only for release branches to avoid spurious merge commits.
  • Squash merges (--squash) are preferred in trunk-based dev: one feature = one commit on main, easy to revert.

Watch Outs

  • Never merge with conflicts unresolved. git merge --abort is your escape hatch.
  • Diverged branch + force push = disaster. Prefer git pull --rebase to avoid merge bubbles.

Interview Angle

Amazon L6 asks merge algorithm deep-dive: recursive vs ORT strategy. Google asks about fast-forward vs 3-way trade-offs. Know when --no-ff matters and when it's noise.

Branching Model

A branch is just a movable pointer to a commit. Creating a branch is instantaneous — no file copies. The default branch is main (formerly master). Git's branching model makes it trivial to isolate work, experiment, and collaborate without affecting the mainline.

git branch                # list local branches
git branch -r             # list remote branches
git branch -a             # list all branches

Creating & Switching Branches

Use git branch to create and git switch (or git checkout) to change branches. The -b flag combines both. HEAD tracks which branch you're on. Use git switch - to go back to the previous branch.

git branch feature-x          # create branch
git switch feature-x          # switch branches
git switch -c feature-x       # create + switch
git switch -                  # previous branch

Merging Strategies

A fast-forward merge simply moves the target branch pointer forward — no merge commit. A 3-way merge creates a new commit with two parents when histories have diverged. --no-ff forces a merge commit even when fast-forward is possible. --squash combines all feature commits into one.

git merge feature             # fast-forward or 3-way
git merge --no-ff feature     # force merge commit
git merge --squash feature    # squash commits
git merge --abort             # abort conflicted merge

Conflict Resolution

Conflicts occur when changes overlap. Git marks conflicted files with <<<<<<<, =======, and >>>>>>> markers. Resolve by editing, then git add and git commit. Use git mergetool for GUI-based resolution.

git status                     # list conflicted files
git diff                       # view conflict markers
git add file.txt && git commit # mark resolved
git log --merge                # see conflicting paths

Remote Branches

Remote-tracking branches (refs/remotes/origin/*) are local cached copies of remote state. git fetch updates them. Use git push origin --delete branch to remove a remote branch.

git fetch origin              # update remote refs
git push -u origin feature    # push + set upstream
git branch -d feature         # delete local (safe)
git push origin --delete feat # delete remote
lightbulbDeep Dive & Key Takeawaysexpand_more

Branches are implemented as 41-byte files in .git/refs/heads/ containing a commit SHA. The HEAD file points to the current branch ref. During a merge, Git finds the merge base (common ancestor) using the lowest common ancestor algorithm. Recursive merge (the default) handles criss-cross histories by merging merge bases iteratively. Content conflicts require manual resolution; tree-level conflicts (rename/rename, rename/delete) are rarer but more complex.
