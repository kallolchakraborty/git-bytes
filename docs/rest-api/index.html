GitHub REST API

schedule2 min readsignal_cellular_1_barFoundational
githubapirest

The GitHub REST API provides programmatic access to GitHub resources. Every request flows through authentication, rate limiting, and pagination before reaching the endpoint that serves or mutates data. This guide covers the full request lifecycle, pagination strategies, and webhook integration.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • GET /repos/{owner}/{repo} — read repo metadata.
  • POST /repos/{owner}/{repo}/issues — create issue.
  • PATCH /repos/{owner}/{repo}/issues/{number} — update issue.
  • Headers: Authorization: Bearer <token>, X-GitHub-Api-Version: 2022-11-28.

Staff+ Patterns

  • Use conditional requests (If-None-Match with ETag) to consume zero rate limit on 304 responses.
  • For collections >1000 items, switch to GraphQL for cursor-based pagination — REST caps at 1000 total items.

Watch Outs

  • Secondary rate limits (429) require monitoring Retry-After headers and exponential backoff.
  • Webhooks must respond with 200 within 10 seconds. Use an async queue.

Interview Angle

GitHub L6 asks about pagination trade-offs: page-based is O(n) for deep pages, cursor-based is O(1). Discuss Link header parsing and conditional requests for rate limit optimization.

API Overview & Auth

GitHub's REST API is versioned at https://api.github.com. All requests require authentication — personal access tokens (classic or fine-grained), GitHub Apps, or OAuth tokens. Fine-grained PATs offer scoped permissions per repository, replacing the broad classic token scopes.

Authentication is passed via the Authorization: Bearer <token> header. Unauthenticated requests are heavily rate-limited (60/hour vs 5000/hour with auth). API versioning uses the X-GitHub-Api-Version header; the current version is 2022-11-28.

Pagination Strategies

GitHub REST API uses page-based pagination by default with ?page=N&per_page=100 (max 100 per page, cap at 1000 items total). The Link header provides navigation — rel="next", rel="last", rel="first", rel="prev".

For collections exceeding 1000 items, cursor-based pagination is available on select endpoints using ?cursor=<opaque-token>. Cursors are stable across real-time data mutations, making them ideal for streaming workflows.

Rate Limiting

Primary rate limit: 5000 requests/hour for authenticated users, 60/hour unauthenticated. The X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset headers communicate current status.

Secondary rate limits apply for abusive or burst patterns — these return 403 or 429 with Retry-After. Best practices include conditional requests (If-None-Match/If-Modified-Since) to consume no rate limit on 304 responses.

CRUD Operations

Standard REST verbs map to GitHub resources:

  • GET /repos/{owner}/{repo} — read repository metadata
  • POST /repos/{owner}/{repo}/issues — create an issue
  • PATCH /repos/{owner}/{repo}/issues/{number} — update an issue
  • DELETE /repos/{owner}/{repo}/issues/{number} — delete an issue (on private repos)

The request body is JSON with Content-Type: application/json. Conditional writes use ETag headers to prevent lost updates.

Webhook Integration

Webhooks deliver real-time events to a configured URL when repository events occur. Each webhook includes an event type header (X-GitHub-Event), delivery GUID (X-GitHub-Delivery), and HMAC-SHA256 signature (X-Hub-Signature-256) for verification.

Payloads are JSON with the full event object. Best practices include: verifying signatures using your secret token, responding with 200 within 10 seconds, and using a queue for async processing. Re-delivery uses exponential backoff with up to 3 attempts.

lightbulbDeep Dive & Key Takeawaysexpand_more

The GitHub REST API is the foundational interface for programmatic interaction with GitHub. It exposes every GitHub resource — repositories, issues, pull requests, actions, projects — through predictable URL patterns and standard HTTP verbs.

Authentication Best Practices

Fine-grained PATs (introduced 2022) are the recommended authentication method. They limit tokens to specific repositories, permissions, and expiration dates. For automation across organizations, GitHub Apps provide installation-level tokens with webhook-driven permissions.

Pagination Deep Dive

When consuming paginated resources, always parse the Link header rather than constructing URLs manually. Libraries like Octokit handle this automatically. For high-volume data extraction, use the GraphQL API which provides cursor-based pagination natively with first/last and after/before arguments.

Rate Limit Optimization

Conditional requests are the most effective optimization: send the If-None-Match header with the last known ETag, and the API returns 304 (Not Modified) consuming zero rate limit if the resource hasn't changed. Similarly, If-Modified-Since with a timestamp achieves the same effect.

Webhook Security

Always validate webhook payloads using the HMAC-SHA256 signature. The secret should be stored in GitHub Secrets or your secret management system. Set a reasonable SECRET_TOKEN and verify with crypto.createHmac('sha256', secret).update(payload).digest('hex').

Code Examples

# List issues with pagination and conditional request
curl -H "Authorization: Bearer $GH_TOKEN" \n  -H "Accept: application/vnd.github+json" \n  -H "X-GitHub-Api-Version: 2022-11-28" \n  -H "If-None-Match: W/\"abc123\"" \n  "https://api.github.com/repos/owner/repo/issues?per_page=100"

# Create an issue with labels and assignees
curl -X POST \n  -H "Authorization: Bearer $GH_TOKEN" \n  -H "Content-Type: application/json" \n  -d '{"title":"Bug: login fails on Safari","body":"Safari 17 breaks OAuth flow","labels":["bug","browser"],"assignees":["kallol"]}' \n  "https://api.github.com/repos/owner/repo/issues"

# Verify webhook signature
echo -n "payload" | openssl dgst -sha256 -hmac "secret"
