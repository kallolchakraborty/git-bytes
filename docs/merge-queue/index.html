Advanced Merge Strategies

schedule2 min readsignal_cellular_4_barAdvanced / Staff+
staff+gitmergefaang

Master GitHub Merge Queue architecture, semantic conflict detection, and the merge algorithms used by FAANG engineering teams to maintain trunk stability at scale.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • git merge --squash feature — squash all feature commits into one.
  • git merge --no-ff feature — force merge commit to preserve branch topology.
  • git merge-base --all main feature — find the merge base(s) for criss-cross merges.
  • git merge-tree main feature — preview merge result without actual merge (Git 2.34+).

Staff+ Patterns

  • Use a merge queue to prevent CI skew: PRs are batched and validated together before landing. GitHub's merge queue (since 2023) creates a temporary branch, runs CI on the combined diff, then fast-forwards main.
  • For the merge-ort strategy (Git 2.34+): it avoids index file locking issues in large repos and provides better conflict messages.

Watch Outs

  • Squash merge loses individual commit authorship. Use --squash only for feature branches, not for shared branches.
  • Merge queues add latency. Set a reasonable queue timeout and have a bypass for hotfixes.

Interview Angle

Amazon L6 asks about merge algorithm internals. Know: recursive merge (default) vs ort (new), rename detection (50% similarity threshold), and the LCA computation for criss-cross histories. The merge queue is a distributed coordination problem.

Merge Queue Architecture

GitHub Merge Queue (GA since 2023) maintains trunk stability by batching PRs, running CI on the combined result, and merging only when green. Architecture: Queuing — PRs enter queue order or priority. Batching — system detects non-conflicting PRs via graph analysis. Integration build — CI runs on squash-merged integration branch. Merge — if green, commits are fast-forwarded to main. Failure recovery — queue drops failing PR, rebuilds remainder.

language-bash
# GitHub Merge Queue configuration
# .github/merge-queue.yml
group:
  pattern: "batch-*"
  batch_size: 5
  batch_timeout: 3600  # 1 hour

check_rules:
  - name: "build"
    type: "checks"
    skip_broken: false
  - name: "test"
    type: "checks"
  - name: "code review"
    type: "approvals"
    min_approvals: 1

# Using Merge Queue via GH CLI
gh pr merge --auto --subject "feat: add OAuth" --body "Closes #123"

# Check queue status
gh pr view --json mergeQueue

# Manual queue interaction
git checkout main
git merge-queue squash feature-branch

Semantic Conflicts

Beyond textual merge conflicts: semantic conflicts occur when code compiles but behaves incorrectly — changed function signatures, renamed APIs, modified data schemas. Detection strategies: run full test suite on merge result, use static analysis (type checking), build graph analysis (Bazel affected tests), and integration testing. Google's Critique and Meta's Phabricator detect semantic conflicts by analyzing dependency graphs.

language-bash
# Semantic conflict detection techniques

# 1. Run affected tests (Bazel)
bazel test $(bazel query 'kind("test", rdeps(//..., //src/changed:target))')

# 2. TypeScript type checking on merge
npx tsc --noEmit --skipLibCheck

# 3. Python mypy on changed files
git diff --name-only main...HEAD | xargs mypy

# 4. GraphQL schema check (schema diff)
npx graphql-inspector diff schema-v1.gql schema-v2.gql

# 5. Protobuf compatibility check
buf breaking --against .git#branch=main

# Manual semantic conflict detection
# Monorepo: run all tests that could be affected
git log --name-only HEAD..origin/main | sort -u | xargs -I{} dirname {} | sort -u | xargs npm test

Merge Algorithms

Git supports multiple merge strategies: recursive (default) — finds merge base, creates virtual ancestor for criss-cross histories. resolve — simpler, faster for two-head merges. octopus — merges >2 heads at once for topic branch consolidation. ours — keeps our version completely. subtree — merges with prefix shifting. ort — the new recursive rewrite (Git 2.33+) using RFC 3 merge, faster and conflict-free rename handling.

language-bash
# Merge strategies
git merge feature              # recursive (default)
git merge -s resolve feature   # resolve (simpler, faster)
git merge -s octopus topic1 topic2 topic3  # octopus (N-way)
git merge -s ours feature      # keep our version

# Merge strategy options
git merge -X theirs feature    # auto-resolve conflicts with theirs
git merge -X patience feature  # use patience diff algorithm
git merge -X diff-algorithm=histogram feature  # histogram diff

# ORT strategy (Git 2.33+)
git merge -s ort feature       # new recursive implementation
# Benefits: better rename detection, no index file needed, faster

# Custom merge driver
echo "*.cfg merge=custom-config" >> .gitattributes
git config merge.custom-config.driver ./merge-config.sh

Automated Policies

Enforce merge policies with GitHub branch protection and custom automation: Linear history — require rebase or squash merges. Signed commits — GPG or SSH signing required. Status checks — CI, lint, test, coverage, security scan. Up-to-date branches — require branch to be current with main. Review requirements — N approvals, code owner review. Auto-merge with conditions — merge when all checks pass.

language-bash
# Branch protection rules (GitHub API)
# Require:
# - Linear history (no merge commits)
# - Required status checks (CI, lint, test)
# - Up-to-date before merging
# - Pull request reviews (1+ approval)
# - Code owner review
# - Signed commits

# Auto-merge GitHub Action
name: Auto Merge
on:
  pull_request:
    types: [labeled]
jobs:
  auto-merge:
    if: contains(github.event.pull_request.labels.*.name, 'auto-merge')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: gh pr merge --auto --squash "$PR_URL"
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

# Merge policy as code (graphql)
mutation {
  updateBranchProtectionRule(input: {
    repositoryId: "...",
    requiresLinearHistory: true,
    requiresSignedCommits: true,
    requiredApprovingReviewCount: 2
  }) { clientMutationId }

FAANG Practices

Google: Piper's 'presubmit' pipeline — every change runs full test suite in a sandbox before landing. Meta: Phabricator + Sandcastle — diff-based CI with incremental builds; auto-reverts on test failure. Microsoft: VFS for Git + Scalar — virtual filesystem with on-demand file fetch; merge queue via Azure DevOps. Amazon: BRAZIL — monorepo with build dependency graph; automatic merge ordering by dependency.

language-bash
# Google's presubmit pipeline (simplified)
# 1. Developer creates CL (changelist)
# 2. Critique runs all affected tests in sandbox
# 3. +2 code review + CI green
# 4. Submitted to Piper trunk
# 5. Piper auto-builds and may auto-revert on rollback

# Meta's Sandcastle flow
git checkout -b my-diff
git commit -m "my change"
git push origin HEAD:refs/diff/my-diff
# Sandcastle runs tests, Phabricator reviews
# Landing: arc land (squash + rebase + push)

# Microsoft's Scalar merge flow
scalar clone <url>
git checkout -b feature
git commit -m "change"
git push origin feature
# PR → Azure DevOps merge queue → CI → merge
lightbulbDeep Dive & Key Takeawaysexpand_more

At FAANG scale, merge strategies directly impact developer productivity and trunk stability. Google's Piper uses a custom VCS with transactional commits — there's no 'merge': changes are applied atomically to the tree. Meta's Mononoke (Git-compatible) supports pushrebase — rebase on push for linear history. GitHub Merge Queue implements a 'merge train' pattern: batches of PRs are tested together, and if any fail, the culprit is ejected and the remaining PRs retry. Semantic conflicts are the hardest to catch — they require type checking, static analysis, and full integration tests. The best merge strategy is the one your team understands and enforces consistently. For small teams (<10), squash-merge with linear history is ideal. For medium teams (10-100), add merge queues. For large orgs (100+), invest in custom tooling and dependency analysis.
