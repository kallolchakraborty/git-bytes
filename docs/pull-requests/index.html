Pull Requests & Code Review

schedule1 min readsignal_cellular_1_barFoundational
githubpull-requestscode-review

Pull requests are the collaboration unit of GitHub — proposing changes, reviewing code, and merging with confidence.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • gh pr create --title "feat: x" --body "Closes #12" — create PR from CLI.
  • gh pr merge --squash — merge with squash (default for trunk-based).
  • gh pr review --approve — approve from CLI.

Staff+ Patterns

  • Squash-merge feature branches to main for a linear history that's revert-friendly. Use merge commits for release branches to preserve topology.
  • Set up a merge queue to prevent CI skew: PRs are validated together before landing, preventing "green on its own, red together" failures.

Watch Outs

  • Merge conflicts in the merge queue require rebasing the PR and re-enqueuing. Communicate this to the team to avoid frustration.
  • Draft PRs still trigger CI. Use [skip ci] in commit messages for docs-only changes.

Interview Angle

Netflix/LinkedIn L6 asks about merge queues and CI at scale. Explain the "social merge" problem: how do you prevent conflicting PRs from breaking main? The answer is a merge queue with batch validation and dependency tracking.

PR Lifecycle

A PR is born from a branch push. Open a PR with a title, description, and optional template. Draft PRs (marked with WIP: or toggle) prevent premature merging. Ready for review changes status. Merge closes the PR. Use gh pr create from the CLI.

gh pr create --title "feat: add login" --body "Closes #12"
gh pr create --draft
gh pr ready                # mark draft as ready
git push --delete origin feature  # cleanup after merge

Review Process

Reviewers can comment, approve, or request changes. Each review creates a summary with line-specific comments. Pull request reviews are grouped into a review conversation. Required reviewers (set in branch protection) must approve before merging.

# PR review states
Comment — general feedback
Approve — ready to merge
Request Changes — must re-review
Dismiss — stale review (admin)

Merge Strategies

Three strategies: Merge commit preserves all commits with a merge bubble. Squash collapses all commits into one — clean but loses granularity. Rebase linearizes history but rewrites hashes. Configure per-repo in Settings > Merge button. The merge queue option adds CI gating.

# gh CLI merge strategies
gh pr merge --merge         # merge commit
gh pr merge --squash       # squash
gh pr merge --rebase       # rebase

Status Checks

Status checks report CI/tool results to PRs: pending, success, failure, error, or neutral. Required checks must pass before merging. Statuses come from commit statuses API or check runs. Use --required-statuses in branch protection.

# Common status checks
continuous-integration/github-actions
codecov/project
WIP — prevents merge with WIP title

Draft PRs

Draft PRs signal work-in-progress. They can't be merged until marked ready. Drafts can still receive comments and CI runs. Use for early feedback or collaborative development. Draft + gh pr create --draft. Convert any PR to draft.

gh pr create --draft
# Mark ready via UI or API
gh pr ready
lightbulbDeep Dive & Key Takeawaysexpand_more

Pull requests are GitHub's implementation of Git's diff and merge machinery wrapped in a collaboration layer. The PR diff is computed server-side using git merge-base to find the fork point, then git diff <base>...<head> for the change summary. GitHub's review system stores reviews as PR review objects with a state (APPROVED, CHANGES_REQUESTED, COMMENT, DISMISSED) and a body. Each review contains pull request review comments (line-specific). The merge queue feature serializes PR merges with CI checks — PRs enter a queue, are validated together, and merged sequentially. PRs can close issues via body keywords (close, closes, closed, fix, fixes, fixed, resolve, resolves, resolved) followed by #issue.
