Actions Runner Architecture

schedule3 min readsignal_cellular_4_barAdvanced / Staff+
devopsactionsrunnerstaff+

Deep dive into GitHub Actions Runner internals: job scheduling, self-hosted runner architecture, autoscaling with ephemeral runners, and network/security considerations at scale.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • ./config.sh --url https://github.com/org --token $TOKEN — register self-hosted runner.
  • runs-on: [self-hosted, linux, x64, gpu] — target specific runners with labels.
  • ACTIONS_RUNNER_DEBUG: true — enable runner-level debug logging.

Staff+ Patterns

  • Use ephemeral runners for security: each job gets a fresh VM that's destroyed after use. Prevents credential leakage between jobs.
  • Set up auto-scaling with actions/actions-runner-controller (ARC) on Kubernetes — scale to zero when idle, scale up on demand.

Watch Outs

  • Self-hosted runners on the same network as production are a security risk. Isolate in a separate VPC with outbound-only access to GitHub.
  • Runner registration tokens expire after 1 hour. Use the actions/runner-registration action for automated provisioning.

Interview Angle

Netflix L6 asks about runner architecture: how does the Actions coordinator dispatch jobs? Discuss polling vs push model, runner group isolation, and the trade-off between hosted (no ops, cost per minute) and self-hosted (free compute, ops cost).

Runner Architecture

Each runner instance runs a Listener process that polls for jobs and a Worker process that executes one job at a time. The listener manages keep-alive, job assignment, and graceful shutdown.

language-text
# Runner directory structure
.
├── bin/
│   ├── Runner.Listener   # polls GitHub for jobs
│   └── Runner.Worker     # executes job steps
├── .runner               # registration credentials
├── .credentials          # OAuth token (scoped)
├── .env                  # environment overrides
├── _diag/                # diagnostic logs
├── _work/                # job working directories
│   └── <repo>/
│       └── <run-id>/
│           └── <job-name>/
# logs flow: _diag/procfilename.log-<pid>.log

Job Scheduling

The Actions backend dispatches jobs to runners based on labels, concurrency limits, and queue depth. Each job gets an isolated working directory and fresh environment with context injection.

language-text
# Job dispatch flow
# 1. Workflow trigger → check-runs queued
# 2. Matrix expansion → N individual job runs
# 3. Label matching: runs-on: [self-hosted, linux, x64]
# 4. Runner picks job via long-poll
# 5. Worker downloads job payload
# 6. Steps execute sequentially
# 7. Logs streamed via chunked upload
# 8. Artifacts uploaded → job marked complete
# Runner labels define routing:
#   runs-on: [self-hosted, linux, gpu]
#   → only runners with ALL three labels qualify

Self-Hosted Runners

Self-hosted runners give you control over hardware, installed software, and network access. They register at the repository, organization, or enterprise level and can be persistent or ephemeral.

language-text
# Register a self-hosted runner
# https://github.com/<org>/<repo>/settings/actions/runners
./config.sh --url https://github.com/<org> \
  --token AAAAAAA --name runner-prod-01 \
  --labels prod,linux,x64 --ephemeral

# Ephemeral mode: runs one job then exits
# Replace token per registration
# systemd service recommended:
./svc.sh install
./svc.sh start

# Enterprise-level registration:
./config.sh --url https://github.com/enterprises/<enterprise> \
  --token AAAAAAA --ephemeral

Autoscaling

Scale sets using the actions/actions-runner-controller or cloud-native auto-scaling groups dynamically provision ephemeral runners based on job queue depth. Scale to zero when idle saves cost.

language-text
# KubeRnetes ARC (Actions Runner Controller)
apiVersion: actions.summerwind.dev/v1alpha1
kind: RunnerDeployment
metadata:
  name: example-runner
spec:
  replicas: 2
  template:
    spec:
      organization: my-org
      labels:
        - linux
        - x64
      ephemeral: true
---
apiVersion: actions.summerwind.dev/v1alpha1
kind: HorizontalRunnerAutoscaler
metadata:
  name: example-autoscaler
spec:
  scaleTargetRef:
    name: example-runner
  minReplicas: 0
  maxReplicas: 20
  scaleUpTriggers:
    - githubEvent:
        workflowJob: {}
      duration: 30m

Network & Security

Self-hosted runners require outbound HTTPS access to api.github.com and blob storage. For private networking, use VPC injection, NAT gateways, and OIDC trust for cloud provider access.

language-text
# Firewall rules for self-hosted runners
# Outbound required:
- api.github.com:443        # job polling
- github.com:443             # checkout
- objects.githubusercontent.com:443  # blob storage
- actions-results-receiver-production.githubapp.com:443  # logs
- vstoken.actions.githubusercontent.com:443  # token exchange

# OIDC trust policy (AWS example):
{
  "Effect": "Allow",
  "Principal": { "Federated": "arn:aws:iam::ACCOUNT:oidc-provider/token.actions.githubusercontent.com" },
  "Action": "sts:AssumeRoleWithWebIdentity",
  "Condition": {
    "StringEquals": { "token.actions.githubusercontent.com:sub": "repo:org/repo:environment:prod" }
  }
}
lightbulbDeep Dive & Key Takeawaysexpand_more

The Actions runner is a .NET application with two primary processes: Runner.Listener handles registration, heartbeats, and job polling via long-lived HTTP connections to GitHub's API. Runner.Worker is spawned per job to execute steps in an isolated _work directory with injected environment variables, secrets, and job context. Self-hosted runners extend this model with custom hardware, software pre-installation, and private network access. The ARC (Actions Runner Controller) project brings Kubernetes-native autoscaling with ephemeral pods that register, execute exactly one job, and terminate — eliminating credential leakage risk and ensuring clean state. For enterprise deployments, runner redundancy, health probes, and deployment group labeling are critical for reliability. Network security requires egress-only access to GitHub endpoints and OIDC-based workload identity federation to avoid long-lived cloud credentials on the runner filesystem.
