Deployment Pipelines

schedule2 min readsignal_cellular_4_barAdvanced / Staff+
devopsdeploymentpipelinesstaff+

Design safe deployment pipelines with environment promotion, approval gates, rollback strategies, and release automation. Covers blue-green, canary, and rolling deploy patterns at scale.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • environment: production — define deployment target with approvals.
  • needs: [build, test] — sequential deployment stages.
  • if: github.ref == 'refs/heads/main' — deploy only from main.
  • rollback: true via git revert + deploy revert commit.

Staff+ Patterns

  • Use progressive delivery: deploy to staging, run integration tests, promote to canary (1%), validate, roll out gradually. Each stage is a separate environment with approval gates.
  • Use deployment events with environment for audit trail — every deploy is recorded in the GitHub API with commit SHA, timestamp, and actor.

Watch Outs

  • Rollback via git revert is safe; git reset --hard on main is catastrophic for collaborators. Always revert.
  • Database migrations can't be reverted by code rollback alone. Use git revert + a compensating migration.

Interview Angle

Netflix L7 asks this directly. Key: design for zero-downtime deployments (blue/green or canary), automatable rollback, and observability during rollout. The deploy should be boring — a script that runs the same way every time.

Environments

GitHub Environments provide protection rules, secret scoping, and deployment tracking. Each environment can require reviewers, wait timers, and branch restrictions before deployment proceeds.

language-yaml
# workflow.yml — environment definition
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://app.example.com
    steps:
      - run: ./deploy.sh

# Environment protection via GitHub UI:
# Settings → Environments → production
# Required reviewers: 2
# Wait timer: 30 minutes
# Deployment branches: main
# Secret isolation:
#   production → PROD_API_KEY
#   staging → STAGING_API_KEY

Approval Gates

Gate promotions with required reviews, status checks, and environment protection rules. Combine with workflow_dispatch inputs for manual control over promotion parameters.

language-yaml
jobs:
  promote-to-canary:
    environment: canary
    runs-on: ubuntu-latest
    steps:
      - run: ./promote.sh --percentage 10

  promote-to-prod:
    needs: promote-to-canary
    environment:
      name: production
      url: ${{ steps.deploy.outputs.url }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/github-script@v7
        id: deploy
        with:
          script: |
            // add deployment status
            await github.rest.repos.createDeploymentStatus({
              ...context.repo,
              deployment_id: process.env.DEPLOYMENT_ID,
              state: 'success',
              environment_url: 'https://app.example.com'
            })

Rollback Strategies

Rollback via git revert and redeploy, or use workflow_dispatch with a target version input. Store deployment metadata in deployment statuses and environment variables for traceability.

language-yaml
name: Rollback
on:
  workflow_dispatch:
    inputs:
      target-version:
        description: 'Git SHA or tag to rollback to'
        required: true

jobs:
  rollback:
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ inputs.target-version }}
      - run: |
          echo "Rolling back to ${{ github.event.inputs.target-version }}"
          ./deploy.sh
      - uses: actions/github-script@v7
        with:
          script: |
            await github.rest.repos.createCommitComment({
              ...context.repo,
              commit_sha: context.sha,
              body: 'Rollback to ${{ inputs.target-version }}'
            })

Blue-Green & Canary

Blue-green deploys maintain two full environments with instant cutover. Canary releases gradually shift traffic while monitoring error rates. Both require a load balancer or service mesh integration.

language-yaml
# Blue-green deploy with AWS ECS
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - id: deploy
        run: |
          aws ecs update-service \
            --cluster prod \
            --service app \
            --task-definition app:${{ github.sha }} \
            --network-config "awsvpcConfiguration={subnets=[subnet-abc]}"
      - id: swap
        run: |
          aws ecs update-service \
            --cluster prod \
            --service app-blue \
            --task-definition app:latest
          # DNS or ALB switch for instant cutover

# Canary with flag-based routing
      - run: |
          # Deploy to 5% of instances
          kubectl set image deployment/app \
            app=myapp:${{ github.sha }} \
            --record
          kubectl scale deployment/app-canary --replicas=1
          # Monitor for 10 minutes
          sleep 600
          # Ramp to 50%
          kubectl scale deployment/app-canary --replicas=5
          sleep 600
          # Full rollout
          kubectl scale deployment/app --replicas=10

Release Automation

Automate semantic versioning, changelog generation, and GitHub Releases with actions/github-script, ncipollo/release-action, and conventional commits parsing for full GitOps workflows.

language-yaml
jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - id: tag
        run: |
          # auto-increment semver from tags
          LATEST=$(git tag -l 'v*' | sort -V | tail -1)
          NEW_VERSION=v$(echo $LATEST | awk -F. '{$NF+=1; print}' OFS='.')
          git tag $NEW_VERSION
          git push origin $NEW_VERSION
          echo "version=$NEW_VERSION" >> $GITHUB_OUTPUT
      - uses: ncipollo/release-action@v1
        with:
          tag: ${{ steps.tag.outputs.version }}
          generateReleaseNotes: true
          makeLatest: true
lightbulbDeep Dive & Key Takeawaysexpand_more

Deployment pipelines at scale require progressive exposure and safety mechanisms. GitHub Actions environments map to deployment targets with branch restrictions, secret isolation, and required reviewers — enabling compliance without leaving the CI/CD platform. For promotion flows, model dev → staging → canary → prod where canary and prod gates require manual approval. Blue-green deployments provide instant rollback via traffic rerouting but double infrastructure cost. Canary releases incrementally shift traffic using weighted load balancer rules, automatically rolling back if error rates spike. Rolling updates replace instances one-by-one — safest for stateless services but slowest during rollback. For rollback automation, run git revert against the deployment commit and push to the protected branch, triggering the same deploy workflow with the prior version. Store deployment IDs, timestamps, and commit SHAs in GitHub Deployment Statuses for full audit trail.
