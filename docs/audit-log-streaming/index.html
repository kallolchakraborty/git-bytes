Audit Log Streaming

schedule1 min readsignal_cellular_4_barAdvanced / Staff+
staff+enterpriseauditsecurity

Audit log streaming forwards GitHub enterprise events to your SIEM in near real-time, enabling security monitoring, compliance reporting, and threat detection.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • gh api /enterprises/{ent}/audit-log -f phrase='created:>2025-01-01' — query audit events.
  • gh api /enterprises/{ent}/audit-log/streams — inspect streaming configuration.
  • POST /enterprises/{enterprise}/audit-log/streams — configure a new stream endpoint.

Staff+ Patterns

  • Stream to AWS S3 with partition prefix (year=2025/month=01/day=15/) for cost-effective long-term storage.
  • Enrich events with SCIM user directory and IP geolocation before feeding to SIEM.
  • Build alerting rules for high-severity events: org ownership changes, audit config changes, mass visibility changes.

Watch Outs

  • Audit log retention is managed by the SIEM/storage, not GitHub. Ensure lifecycle policy archives old data.
  • Webhook delivery has 3 retries with exponential backoff — configure a dead-letter queue for hardened delivery.
  • Filter out noisy events (repo.push, git.clone) to reduce SIEM costs and alert fatigue.

Interview Angle

Snowflake L6: "Design an audit system for a multi-tenant SaaS built on GitHub Enterprise." Discuss: tenant isolation in audit streams, immutable storage for compliance, and real-time alerting on anomalous access patterns (e.g., off-hours admin changes by non-SRE users).

Event Taxonomy

Audit events fall into categories: authentication (login, SSO), repository (create, delete, visibility change), organization (member add/remove, policy change), team (create, membership), security (Dependabot alert, secret scan), and Actions (workflow run, self-hosted runner). Each event has @timestamp, actor, action, resource, and org. Query via the REST API GET /enterprises/{enterprise}/audit-log with filters for phrase, include (git/web/all), and time ranges.

# Fetch audit events for the last hour
gh api /enterprises/my-ent/audit-log \
  --jq '.[] | select(.action == "repo.create")' \
  -f phrase='created:>2025-01-01'

SIEM Integration

Audit log streaming sends events to AWS S3, Azure Blob Storage, Google Cloud Storage, or HTTP endpoints (Splunk HEC, Datadog). Configure streaming at the enterprise level under Monitor > Audit log > Streaming. Events are delivered in JSONL format with batching. Each batch contains multiple events with @timestamp and event_type. Set up a SIEM connector to parse fields into the common schema.

# Splunk HEC configuration
url: https://splunk.company.com:8088/services/collector
  token: ****
  index: github_audit
  sourcetype: github:audit

Webhook Enrichment

Audit log webhooks include action, actor, actor_ip, user_agent, org, repo, and detailed payload fields. Enrich events by joining with SCIM user directory or IP geolocation. Build a serverless function to enrich the webhook payload before forwarding to the SIEM. Store enriched fields as index-time fields for faster search.

# Enriched payload
{
  "action": "repo.create",
  "actor": "jdoe",
  "department": "engineering",
  "geo": {"city": "San Francisco", "country": "US"},
  "risk_score": 10
}

Compliance Reporting

Audit logs support SOC 2, FedRAMP, and GDPR compliance. Retain audit logs for at least 365 days (or as required by your compliance framework). Build dashboards for: privilege escalation (admin changes), unusual access patterns (off-hours logins), and external collaboration (outside collaborator additions). Use the audit-log GraphQL API for custom compliance queries.

query {
  enterprise(slug: "my-ent") {
    auditLog(stream: true, after: $cursor) {
      edges {
        node {
          action
          actor { name }
          occurredAt
        }
      }
    }
  }
}

Alerting Rules

Define alerting rules in your SIEM for high-severity events: org ownership changes, audit log configuration changes, API token creation without expiry, and mass repo visibility changes. Integrate with PagerDuty or Slack via webhooks. Use rate-based alerts for brute force attempts and geo-anomaly detection for impossible travel.

# Example alert: unauthorized access attempt
sourcetype="github:audit" action=login.failed
| stats count by actor_ip
| where count > 10
lightbulbDeep Dive & Key Takeawaysexpand_more

Audit log streaming is the backbone of GitHub enterprise security monitoring. Every event — from repo.create to org.update_member_role — is captured with the actor IP, user agent, and timestamp. Events are streamed via webhooks to configured endpoints or cloud storage. AWS S3 is the most common target: events are delivered as .jsonl files with a path pattern like year=2025/month=01/day=15/. Enable streaming in the enterprise settings or via the API POST /enterprises/{enterprise}/audit-log/streams. Each stream can be configured with enabled_events filters to reduce noise. The audit log API supports the phrase query parameter for full-text search across event fields, with include=git for Git-related events and include=web for UI/API events. Retention is managed by the SIEM, not GitHub — ensure your storage lifecycle policy archives old data. For compliance, extract audit log data daily to an immutable storage bucket (S3 Object Lock / Azure Immutable Blob). Build an alerting pipeline that enriches events with context (user department from SCIM, IP risk score) and routes critical alerts to security team PagerDuty. Monitor streaming health — GitHub reports stream health status in the UI and via the API at GET /enterprises/{enterprise}/audit-log/streams.
