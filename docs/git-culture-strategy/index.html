Git Culture & CI/CD Strategy

schedule1 min readsignal_cellular_4_barAdvanced / Staff+
staff+leadershipculturestrategy

Building a Git-centric engineering culture requires a deliberate maturity model from ad-hoc workflows to optimized, data-driven delivery pipelines.

quick_reference Staff+ Cheat Sheetexpand_more

Core Frameworks

  • CI/CD Maturity Model: Ad Hoc → Standardized → Automated → Optimized.
  • GitHub Flow — single main branch, short-lived feature branches, PR reviews, deploy on merge.
  • Trunk-Based Development — daily merges to main, feature flags, short-lived branches (<48h).
  • Git Flow — develop, release/*, hotfix/* branches for scheduled releases.

Staff+ Patterns

  • Culture change requires top-down support (executive sponsorship) AND bottom-up adoption (champions, guilds).
  • Enforce conventions mechanically (rulesets, linters) rather than through code review — policy as code scales.
  • Invest in the golden path — make the right thing the easy thing. PR templates, auto-linting, one-command deploy.

Watch Outs

  • A single Git workshop is not enough — training must be repeated and tracked.
  • Enforcing trunk-based dev without feature flags leads to broken main and frustrated teams.
  • DORA metrics without context are misleading — a team deploying daily may have high change failure rate.

Interview Angle

Microsoft L7: "You're hired to transform Git culture at a 10K-engineer org. What's your 12-month plan?" Answer: assess maturity (audit repos/branch protection), establish standards (GitHub Flow + rulesets), automate (CI/CD + DORA dashboards), and evangelize (training, guilds, internal showcases).

CI/CD Maturity Model

Level 1 — Ad Hoc: No branching strategy, manual deployments, no CI. Level 2 — Standardized: Established branching model (GitHub Flow), PR reviews required, basic CI (lint + test). Level 3 — Automated: Deploy on merge, automated releases, environment promotion gates, security scanning. Level 4 — Optimized: DORA metrics tracked, deployment frequency measured, continuous improvement, developer productivity data drives tooling investments.

Team Workflows

Define expected workflows for each team type: platform teams (trunk-based, short-lived branches, frequent merges), application teams (GitHub Flow, feature branches, PRs with reviewers), and SRE teams (GitOps, declarative configs in Git, approval gates). Document the expected workflow in a CONTRIBUTING.md and enforce via branch protection rulesets and .github/workflows templates.

Training Programs

Invest in Git training for all engineers. Foundational: git init through rebase and bisect. Intermediate: rerere, worktree, filter-branch (and filter-repo). Advanced: Git internals, custom merge drivers, hooks, and performance optimization. Pair training with internal documentation and gh extension tutorials. Run regular Git office hours for Q&A.

Org Standards

Establish org-wide standards enforced through policy as code: commit message format, branch naming convention (type/ticket-description), PR size limits (<400 lines), required checks before merge, and signed commits. Use .github repositories for default community health files and reusable workflows. Publish a handbook page on Git conventions with examples and rationale.

# Branch naming
feat/AUTH-123-add-oidc
fix/BILL-456-rounding-error
chore/deps-upgrade-express

Measuring Adoption

Track adoption of Git culture through metrics: % of PRs following commit conventions, deployment frequency per team, average PR cycle time, % of repos with branch protection, and CI pass rate. Build a dashboard that breaks down by team. Use GitHub API to collect data weekly and trend over time. Celebrate teams that level up with internal showcases.

# Query: repos without branch protection
repos=$(gh api /orgs/$ORG/repos --jq '.[].name')
for r in $repos; do
  gh api repos/$ORG/$r/branches/main/protection \
    --silent || echo "$r: no protection"
done
lightbulbDeep Dive & Key Takeawaysexpand_more

Git culture is not about using Git — it's about how Git usage enables engineering velocity, quality, and collaboration. The maturity model provides a roadmap: teams at Level 1 need basic Git training and branching strategy adoption; Level 2 requires PR culture and CI basics; Level 3 needs deployment automation and environment management; Level 4 demands continuous improvement based on data. The progression is not purely linear — larger orgs may have teams at different levels, and the platform team should focus on raising the floor. Culture change requires top-down support (executive sponsorship, dedicated time for improvement) and bottom-up adoption (champions, guilds, internal communities). Training should be tracked and repeated — a single Git workshop is not enough. Pair training with hands-on labs in a sandbox environment. Org standards must be enforced mechanically (rulesets, linters) rather than through code review — policy as code is more scalable than manual enforcement. For adoption measurement, look for leading indicators (training completion, PR template usage, commit message formatting) and lagging indicators (deployment frequency, MTTR, developer satisfaction). The ultimate goal is that Git conventions become invisible — developers follow them not because they are enforced but because they are the most natural and efficient way to work. Regular retrospectives on the development workflow itself (meta-retros) surface friction points. Invest in tooling that makes the right thing the easy thing: autocomplete for branch names, default PR templates, automated commit linting, and one-command deployment from a PR.
