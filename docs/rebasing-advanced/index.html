Rebasing & Advanced Git

schedule1 min readsignal_cellular_3_barIntermediate
gitadvancedrebasing

Rebasing rewrites history by replaying commits onto a new base. Master these advanced workflows to maintain a clean, linear project history.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • git rebase -i HEAD~3 — interactive rebase to squash, reword, or drop recent commits.
  • git rebase main — replay current branch on top of main. Keeps history linear.
  • git cherry-pick abc123 — apply a specific commit from another branch.
  • git stash push -m "WIP" — stash with a message for context.
  • git bisect run npm test — automated binary search for bug-introducing commit.
  • git reflog — recovery safety net. Shows every HEAD movement for 90 days.

Staff+ Patterns

  • Interactive rebase before merging: squash fixup commits, reword messages to follow convention, drop debug commits. Your PR should tell a coherent story.
  • Use git rebase --onto to move a branch from one base to another: git rebase --onto main feature~3 feature.

Watch Outs

  • Never rebase a shared branch. If you do, everyone downstream must git rebase --onto or re-clone.
  • Rebase during a bisect can invalidate the bisect state. Always complete bisect before rebasing.

Interview Angle

Meta L6 asks about rebase vs merge trade-offs. The key insight: rebase gives a linear history ideal for git bisect, but loses the context of parallel development. Merge preserves topology. Know both and when to use each.

Interactive Rebase

git rebase -i opens a todo list of commits with operations: pick, reword, edit, squash, fixup, drop. Squash combines a commit into the previous one. Fixup is like squash but discards the message. Edit pauses to let you amend. Reword changes the message.

git rebase -i HEAD~3       # rebase last 3 commits
git rebase -i main         # rebase all commits since main
git rebase --continue      # continue after resolving
git rebase --abort          # abort and restore state

Cherry-Pick

git cherry-pick applies a single commit from another branch onto HEAD. Useful for selective backports. Git computes the patch and applies it; conflicts are resolved the same way as merges. Cherry-picking a merge commit requires -m 1 to specify the mainline parent.

git cherry-pick abc123         # apply commit
git cherry-pick abc123..def456 # range pick
git cherry-pick -m 1 merge~    # cherry-pick merge

Stashing

git stash temporarily shelves changes so you can switch branches. The stash is a stack (LIFO). Use pop to apply and drop, or apply to keep the stash. Stashes can hold staged and unstaged files separately (--staged, --keep-index).

git stash push -m "WIP"      # stash with message
git stash list               # list stashes
git stash pop                # apply + drop latest
git stash apply stash@{2}    # apply specific stash

Bisect Debugging

git bisect binary-searches commit history to find the commit that introduced a bug. Mark a known-good commit and a known-bad commit; Git checks out the midpoint. You mark each as good or bad, and Git narrows down the culprit in O(log n) steps. Automate with git bisect run using a test script.

git bisect start             # start bisect
git bisect bad               # current is bad
git bisect good v1.0         # tag is good
git bisect run npm test      # automated bisect

Reflog Recovery

Git's reflog records every HEAD movement for 90 days (default). It's your safety net — recover seemingly lost commits, reset branches, or undo rebases. The reflog is local and not shared. Entries expire; use git reflog expire to manage them.

git reflog                   # show HEAD history
git reflog show feature      # show branch history
git reset --hard HEAD@{2}    # recover to reflog entry
git reflog expire --expire=now --all
lightbulbDeep Dive & Key Takeawaysexpand_more

The golden rule of rebase: never rebase commits that exist on a shared branch (one pushed to others). Rebase rewrites commit hashes — the new commits have different timestamps, trees, and parent pointers. During rebase, Git uses the same 3-way merge machinery as merges but applies changes one commit at a time. Cherry-pick is implemented as a rebase of a single commit. The reflog is stored in .git/logs/ and is the single most useful recovery tool. Git's auto-garbage-collection prunes unreachable objects; the reflog prevents reachable-from-reflog objects from being GC'd.
