Repositories & Issues

schedule1 min readsignal_cellular_1_barFoundational
githubreposissues

Repositories are the heart of GitHub — every project, every collaboration, every discussion starts here. Issues track work, bugs, and feature requests.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • gh repo create my-proj --public --clone — create repo via CLI.
  • echo "closes #123" >> commit.txt — close issues from commit messages.
  • .github/ISSUE_TEMPLATE/bug_report.yml — structured issue forms.

Staff+ Patterns

  • Use issue templates with YAML forms to enforce structured bug reports — drastically reduces triage time on large projects.
  • Enable branch protection rules requiring signed commits and status checks on main. Always use CODEOWNERS for automatic reviewer assignment.

Watch Outs

  • Webhooks have a 10-second timeout. For long processing, accept immediately and enqueue for background work.
  • Branch protection with "Include administrators" can lock admins out. Have a fallback access path (deploy key, PAT).

Interview Angle

GitHub L6 asks about API pagination and webhook reliability. Know the difference between REST cursor pagination and GraphQL connections. Discuss webhook retry semantics and idempotency keys.

Repository Setup

Create a repo with a README, .gitignore template, and license. Choose between public, private, or internal visibility. Repository templates let you bootstrap new projects with predefined files and structure. Use topics to categorize discoverable repositories.

# Create via CLI (gh)
gh repo create my-project --public --clone
gh repo create --template org/template --public

Issue Tracking

Issues can be created manually, from commit messages, via email, or through issue forms. Markdown supports task lists (- [ ]), @mentions, and cross-references (#123, owner/repo#456). Issue templates standardize bug reports and feature requests. Use close #123 in a commit to auto-close.

# Close from commit
closes #123, fixes #456
# Cross-reference
See also org/repo#789

Labels & Milestones

Labels group issues (bug, enhancement, help wanted, good first issue). Milestones track progress toward a release or sprint with a due date and progress bar. Use priority: critical labels with color coding for triage workflows.

# Default labels
bug (red), enhancement (blue), documentation (green)
duplicate (gray), invalid (pink), question (purple)

Issue Templates

YAML-based issue forms in .github/ISSUE_TEMPLATE/ provide structured inputs: text, textarea, dropdown, checkboxes, markdown. Blank issues can be disabled. Template choosers redirect to external URLs or other templates. Use config.yml for contact links.

.github/ISSUE_TEMPLATE/
  bug_report.yml
  feature_request.yml
  config.yml           # contact links + blank issue policy

Repository Settings

Settings control: merge strategies (merge commit, squash, rebase), branch protection rules (require PRs, status checks, signed commits), autolink references, push restrictions, and deploy keys. Webhooks send JSON payloads on events (push, issues, PRs).

# Branch protection rules
Require PR reviews (1-6 approvers)
Dismiss stale reviews
Require status checks (CI must pass)
Require signed commits
lightbulbDeep Dive & Key Takeawaysexpand_more

GitHub issues are backed by the GraphQL API — each issue is a node in the global object graph with connections to labels, assignees, milestone, project cards, timeline events, and comments. Issue events (labeled, assigned, closed, cross-referenced) are stored as timeline items and accessible via the API. The issue number is unique per repository, not globally. Repository settings are stored in the GitHub database alongside the git data. Branch protection rules are enforced server-side: pushes directly to protected branches are rejected by the Git server before any hooks run. Webhooks have a 10-second timeout; use X-Hub-Signature-256 for secret validation.
