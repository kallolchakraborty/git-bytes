Staff+ Interview Questions

Staff+ engineering interviews assess system design, architectural thinking, and organizational impact through open-ended problems drawn from Git and GitHub scale challenges. Below are commonly asked questions at each FAANG+ company with the position level they target.

quick_reference Staff+ Cheat Sheetexpand_more

Core Frameworks

  • STAR — Situation, Task, Action, Result. Structure for behavioral answers.
  • System design dimensions: data partitioning, consistency vs availability, protocol design, performance at scale.
  • Company-specific patterns: Google (distributed systems), Meta (performance, monorepo), Amazon (Dive Deep), Microsoft (Scalar/VFS), Netflix (CD/canary).

Staff+ Patterns

  • Staff+ answers demonstrate: org-wide scope, technical depth with trade-offs, influencing without authority, mentorship, and learning from failure.
  • For system design interviews: always discuss trade-offs (consistency vs availability, speed vs safety) and reference real systems (Google Piper, Meta Mononoke, Microsoft Scalar).
  • Draw the commit graph — visualize what happens during merge, rebase, cherry-pick. Shows deep understanding.

Watch Outs

  • Common L4/L5 mistakes: confusing reset --soft/mixed/hard, not knowing git push internals, unable to recover from force-push.
  • For Staff+: don't just describe, evaluate. "We chose X over Y because of Z trade-off."
  • Behavioral answers must show org-wide impact, not just team-level. Use the STAR format with metrics.

Interview Angle

This whole guide is the interview angle. Key preparation: git bisect run scripting, git filter-repo for history rewriting, partial clone architecture, merge queue design, and familiarity with each FAANG company's specific Git infrastructure and interview themes.

Google L6: Git Hosting System Design

Problem: Design a Git hosting service for 100M repositories serving 50M developers. Discuss data partitioning (repos across shards), Git protocol handling (SSH vs HTTP), object storage (content-addressable blobs in S3), reference resolution (refs in DB), and availability vs consistency trade-offs. Key considerations: pack file generation, delta compression, and garbage collection at scale.

At Google: Interviewers look for familiarity with Piper's architecture. Key insight: separate control plane (auth, repo routing) from data plane (object storage). Google asks this as a general distributed systems question — you're expected to design from first principles. Mention: consistent hashing for repo sharding, packfile bitmaps for clone speed, and multi-tier cache (L1: Redis, L2: SSD, L3: S3).

Meta L6–L7: Monorepo Architecture for 10K Engineers

Problem: Design a monorepo for 10K developers with 1M files. Discuss partial clone, sparse checkout, git fsmonitor (file system monitor), and shallow fetch strategies. How does git clone --filter=blob:none differ from --depth 1? Design the CI strategy: only build changed projects via dependency graph analysis. How do you scale git status performance?

At Meta: Mononoke is Meta's Git-compatible server. They ask this to see if you understand why Mercurial was chosen over Git (better Python API for custom tooling). Key insight: Meta's watchman integration with git fsmonitor eliminates lstat() for status. Sparse checkout is essential — 90% of engineers only touch 10% of the codebase. For CI, Meta uses hg sparse equivalent to build only affected targets.

Amazon L6 (SDE III): Merge Algorithm Deep Dive

Problem: Explain how Git's merge algorithm works. Walk through recursive merge strategy, merge base computation (lowest common ancestor), three-way merge, conflict detection with common ancestor, and rename detection (similarity index). Discuss alternatives: ours, theirs, octopus, and subtree merges. When does the recursive strategy fail, and how does the ort strategy improve it?

At Amazon: This tests Dive Deep leadership principle. Amazon expects you to know the internals: merge-recursive uses a recursive LCA computation for criss-cross merges, while merge-ort (added in Git 2.34) does it in O(n) time with explicit conflict labels. Mention rename detection threshold (50% similarity by default), the diff.renameLimit config, and how ort avoids index file locking issues in large repos.

Microsoft L7: Scalar / VFS for Git Design

Problem: Git operations on the Windows repository (300GB+, 3000+ engineers) take minutes. Design a solution that makes git status, git checkout, and git fetch complete in under 1 second. How does the Git Virtual File System (VFS) work? What does Scalar (successor to GVFS) do differently?

At Microsoft: This is the defining question for the Azure DevOps / VFS team. Key components: (1) Virtual filesystem using projected filesystem (ProjFS on Windows, FUSE on Mac) — only materialize files on first access. (2) Background prefetch of packfiles. (3) Git protocol extension: --filter=tree:0 to defer all blob fetching. (4) Multi-pack index (MIDX) for O(1) object lookups. (5) Scalar's improvements: no custom filesystem driver needed, uses git sparse-checkout and git fsmonitor--daemon instead. Mention the git maintenance background job and prefetch task.

Netflix L6: Actions Scale & Deployment Pipeline

Problem: GitHub Actions processes 10M workflow runs per day. Design the architecture: webhook ingestion, workflow parsing, job scheduling on self-hosted/ephemeral runners, artifact storage, and log streaming. Discuss runner autoscaling, concurrency limits (concurrency key), caching (actions/cache), and the matrix strategy. How would you reduce queue time during peak hours?

At Netflix: Focus on CD (continuous deployment) — every commit goes to production. Netflix asks: "How do you ensure Git tags are cryptographically signed before deploying?" "Design a canary pipeline where only 1% of traffic hits the new commit." Discuss: Spinnaker integration, manual approval gates, deployment strategies (rolling, blue/green). Key insight: Netflix's concurrency key on workflow level prevents overlapping deployments to the same environment. Mention github.event.release trigger for release-based pipelines.

Apple L7: Custom Merge Driver & Xcode Integration

Problem: The Xcode project file (.pbxproj) is a plist with UUIDs that change on every file add/remove — causing constant merge conflicts. Design a custom merge driver and a workflow that minimizes developer friction.

At Apple: This is asked verbatim by Xcode infrastructure teams. Requirements: (1) A merge driver that sorts UUIDs, de-duplicates entries, and handles parallel edits. (2) A hook that warns developers when they're about to commit a malformed project file. (3) A server-side check that rejects PRs breaking the project structure. Bonus: design a replacement format for .pbxproj that's merge-friendly (e.g., directory-based project format like Swift Package Manager). Apple also asks about git worktree for building multiple branches simultaneously (Intel + Apple Silicon builds).

Stripe L6: Server-Side Hook Architecture

Problem: Design a system of server-side Git hooks that enforces compliance policies (commit message format, signed commits, file size limits, secret detection) across all repositories in an organization, without slowing down legitimate pushes.

At Stripe: Reliability is paramount — hooks must not be a bottleneck. Design: (1) Pre-receive hook parses updated refs, runs lightweight checks synchronously. (2) Heavy checks (secret scanning, large file detection) run asynchronously via a job queue, with a status check API. (3) Use git cat-file --batch for efficient object scanning. (4) Caching: skip checks on commits already scanned. (5) If async check fails, create a check-run failure — don't block the push. Stripe's engineering culture values pragmatic trade-offs — discuss the latency vs safety trade-off of synchronous vs async checks.

Uber L6: Microservices Monorepo Tooling

Problem: Design the Git tooling and workflow for a monorepo with 2000 microservices owned by 50+ teams. Each service has its own CI pipeline, deployment schedule, and ownership. How do you ensure teams can work independently without stepping on each other?

At Uber: Key challenges: (1) CODEOWNERS at scale — per-service ownership with auto-requested reviews. (2) CI — build only services affected by a change (git diff --name-only + dependency graph). (3) Deployment — each service deploys independently via its own workflow file. (4) Pre-commit testing — run service-specific tests before merge. Uber's solution: go/x monorepo with buf for protos, pants/bazel for builds. Discuss git sparse-checkout cone mode so engineers only check out their service's code.

Snowflake L6: Perforce-to-Git Migration

Problem: Migrate a 15-year-old Perforce depot with 100K changelists and complex branch mappings to Git while preserving all history, commit metadata, and branch relationships. The team is 500 engineers who must not lose productivity during the migration.

At Snowflake: This is a real problem — Snowflake migrated from Perforce to Git. Key steps: (1) Map Perforce branches to Git branches using git p4 with custom branch mappings. (2) Preserve commit timestamps, authors, and change list numbers as git notes. (3) Handle Perforce's p4 move semantics (Git auto-detects renames via similarity). (4) Migrate in phases: first 6 months in read-only Git + Perforce write, then full cutover. (5) Provide training and migration guides. Discuss: git filter-repo for splitting history, git lfs for large binaries, and rollback plan (keep Perforce archive for 1 year).

GitHub L6: API & GraphQL Design

Problem: Design the GitHub REST API and GraphQL API for listing commits on a repository with 1M+ commits. How do you paginate efficiently? How do you support filtering by author, date range, and file path? How does the GraphQL API differ from REST in terms of performance and developer experience?

At GitHub: The GitHub API team asks this. Key points: (1) REST pagination uses page-based (page/per_page) — efficient for random access but slow for deep pages. (2) GraphQL uses cursor-based pagination (first/after) — O(1) per page regardless of depth. (3) Commit listing uses git rev-list with --skip for page offset — but skip goes O(n). Better: use --after and --before with commit timestamps for cursor-based navigation. (4) For file-path filtering, git rev-list -- walks the commit graph. (5) For performance: commit-graph speeds up reachability; bitmap indexes speed up clone/fetch but not rev-list — a custom commit index is needed for fast pagination.

LinkedIn L6: Code Review Infrastructure at Scale

Problem: Design the code review infrastructure for 25M engineers across 200M repositories. How do you recommend reviewers, detect similar PRs, merge conflicting PRs in parallel, and provide meaningful review analytics? How does the merge queue work when 1000 PRs target the same branch?

At LinkedIn: This question tests your understanding of code review as a distributed coordination problem. Key components: (1) Reviewer recommendation using collaborative filtering — if A and B reviewed the same files, recommend B when A is busy. (2) Merge queue with dependency tracking — PRs that touch disjoint files merge in parallel; PRs with overlapping files are serialized. (3) Review analytics: time-to-first-review, re-review rate, and reviewer workload balancing. LinkedIn uses trrs (turbo repo review system) internally. Discuss the trade-off between merge velocity (merge as fast as possible) and merge safety (never break main). Git's merge-base is computed per-PR — if another PR merged while yours was in review, the merge queue re-validates against the new base. Mention git merge-tree for previewing merge results without actually merging, and how GitHub's merge queue (since 2023) batches PRs and runs CI on the combined diff.

Airbnb L6: Service Extraction from a Monorepo

Problem: Your monorepo has grown to 50 microservices. The payments team wants to extract their service into a separate repo so they can ship independently. Design the extraction: preserve full Git history for the payments directory, maintain backward compatibility for consumers, and set up cross-repo dependency management. How do you handle shared protobuf definitions and common libraries?

At Airbnb: This is based on Airbnb's real migration from monorepo to polyrepo. Key steps: (1) Use git filter-repo with --path payments/ to split history — this rewrites commits to only include the payments directory, preserving author, timestamps, and messages. (2) Subtree replacement: keep a git subtree in the monorepo pointing to the new repo for a transition period. (3) Shared protos go into a third protos repo consumed by both — versioned with tags like protos/v1.2.3. (4) Common libraries become internal packages published to a private registry. Key insight: Airbnb used a dual-write phase where both monorepo and polyrepo accepted changes for 3 months, with a bot that synced commits in both directions. Discuss the trade-off: monorepo gives atomic cross-service changes but requires tooling investment; polyrepo gives team autonomy but requires dependency versioning discipline. Mention tools like git subtree split and git filter-repo --refs for the extraction itself.

DoorDash L6: Multi-Repo Dependency Management

Problem: Your organization has 200 microservices across 200 repos. Service A depends on B v2.1, and B depends on C v3.0. When C releases v3.1, you need to update B then A. Design a system that tracks these dependencies, automates upgrade PRs, and notifies teams when they're blocking downstream consumers.

At DoorDash: This tests your understanding of dependency management at scale. Key components: (1) A dependency graph stored as a DAG (directed acyclic graph) — each node is a repo, each edge is a version constraint. (2) Automated dependency update bot — when C publishes v3.1, the bot creates a PR in B to update the dependency, then after B's PR merges, creates a PR in A to update to B's new version. (3) Versioning convention: semver with strict breaking-change detection. (4) Integration testing on the combined upgrade before merging. DoorDash uses Renovate bot extensively. Discuss trade-offs: eager upgrades (always stay on latest — high merge overhead but low technical debt) vs lazy upgrades (only upgrade when needed — lower overhead but risk of cascading outdated dependencies). Mention git submodule (not recommended for this — too manual) vs package managers like npm, go mod, or bazel for automatic resolution. For zero-downtime dependency rollouts, use canary releases: upgrade staging first, run integration tests, then promote to production.

Twitter/X L6: Open Source Workflow on a Public Monorepo

Problem: Your company is open-sourcing a large internal project. Design the contribution workflow: CLA (Contributor License Agreement) signing, automated triage of community PRs, CI for external contributors without exposing secrets, fork management, and maintaining a healthy open source community. How do you handle a PR that's been stale for 6 months?

At Twitter/X: Twitter open-sourced Bootstrap, Finagle, Scalab — all from internal monorepos. Key design: (1) Bot-assisted triage — label PRs by area (bug, feature, docs), auto-respond to first-time contributors with onboarding links. (2) CLA bot — blocks merge until CLA is signed via GitHub check. (3) CI for forks — use pull_request_target event with safe checkout (no write tokens exposed to untrusted code). (4) Stale PR policy — after 30 days of inactivity, bot comments asking for update; after 60 days, closes with "please open a new PR if still relevant." (5) Maintainer rotation — ensure no single bus factor. Twitter's twitter.github.io used Jekyll with community PRs. Key insight: the git workflow for open source is different from internal — forks are shallow clones by default, so maintainers use git fetch upstream pull/ID/head:pr-branch to test PRs locally. Discuss git rebase vs git merge for community PRs: Twitter prefers merge commits to preserve contribution attribution. For internal-to-open-source migration, use git filter-repo --strip-custom-commits to remove internal references before publishing.

Multi-Company L6+: Git Security & Supply Chain

Problem: Design a Git security strategy for a large organization. Cover: signed commits (GPG), signed tags, commit verification badges on GitHub, secret scanning (pre-commit hooks + server-side scanning), branch protection rules, and SBOM generation from Git history. How do you prevent a compromised laptop from pushing malicious code?

At Google/Meta/Microsoft: Supply chain security is a Staff+ topic. Key components: (1) GPG signed commits — enforce with a pre-receive hook that rejects unsigned commits from non-CI users. GitHub shows a verified badge for GPG-signed commits. (2) SSH keys with hardware-backed keys (YubiKey) — integrate with corporate PKI. (3) Secret scanning — truffleHog or git-secrets as a pre-commit hook, plus GitHub's push protection API that scans every push. (4) Branch protection — require signed commits, status checks, linear history, and no force pushes on main. (5) SBOM — git log --format='%H %s' --since=1year to generate a list of dependency changes; combine with syft or trivy for vulnerability scanning. (6) Incident response — git revert a malicious commit atomically across all branches, rotate all secrets in the affected commit using git log --all --diff-filter=A -- '*.env' to find leaked secrets. Google's borg/piper uses cryptographic attestation for every commit — discuss how CLAs and signed commits create an auditable chain of custody. Mention git fsck for repository integrity verification and git rev-list --objects --all | git cat-file --batch-check for object auditing.

Multi-Company L7: Production Incidents & Git-Based Rollback

Problem: A bad deployment goes to production — all pods crash-looping. Design the rollback strategy using Git: how do you revert, how fast can you revert, and how do you prevent the bad commit from being re-deployed? Design a canary deployment pipeline integrated with Git tags. How do you handle a rollback that's needed 3 months after the bad commit (when 100 other commits landed after it)?

At Netflix/Amazon/Google: Incident management at Staff+ level. Key strategies: (1) Immediate rollback: git revert creates a new commit that undoes the change — this is cleaner than git reset --hard because it doesn't rewrite history. Deploy the revert commit. (2) For a deep rollback (3 months later): git revert ..HEAD --no-commit to undo everything since the bad commit, then cherry-pick the good ones back. Use git log --grep to identify which subsequent commits are safe. (3) Tag-based deployment: every deploy creates a Git tag (deploy-prod-v435). Rollback = deploy the previous tag. (4) Canary pipeline: deploy tag to 1% of traffic, run synthetic checks for 5 minutes, auto-promote if healthy, auto-rollback if error rate spikes. GitHub Actions: deployment event with environment: production and required_contexts. (5) Postmortem: use git bisect to find the exact bad commit if the root cause isn't obvious. Netflix's Spinnaker integrates with Git tags natively — every pipeline run is associated with a Git commit SHA. Amazon's CodePipeline uses git commit ID as the source artifact version. Google Cloud Deploy uses skaffold with Git-based delivery pipelines. Key insight: revert is a new commit, not a history rewrite — this preserves the audit trail. For database migrations that can't be reverted with a simple code revert, use git revert + a compensating migration. Discuss the difference between git revert (safe, auditable) and git reset (dangerous, rewrites history).

Multi-Company L6+: Developer Experience & Inner Source

Problem: Your company has 5000 engineers across 200 teams. Most teams don't know what other teams are building, leading to duplicated effort and inconsistent practices. Design an inner source program: how do you encourage cross-team contributions using Git and GitHub workflows? How do you measure developer productivity and satisfaction?

At Microsoft/Google/Stripe: Inner source applies open source practices inside an organization. Key design: (1) Repository discovery — a searchable catalog of all internal repos with README, API docs, and ownership. Use GitHub's repo topics and CODEOWNERS for discoverability. (2) Contribution workflow: external (to the team) contributors fork the repo, create a PR, and the owning team reviews — identical to open source. Use CONTRIBUTING.md and issue templates. (3) Inner source license: a standard internal MIT-equivalent that allows any team to use any repo. (4) Metrics: DORA metrics (deploy frequency, lead time, MTTR, change failure rate), plus contribution metrics (number of cross-team PRs, time-to-first-PR for new hires). (5) Incentives: include cross-team contributions in performance reviews. Microsoft's 1ES (One Engineering System) standardizes Git workflows across all of Microsoft. Google's citC (Clients in the Cloud) uses Piper for the entire monorepo. Stripe's stripe-internal monorepo has contribution guidelines published as a Markdown file at the root. Key insight: inner source fails when the friction to contribute is higher than the benefit — invest in CI/CD that makes it trivial to build and test any repo. Use git sparse-checkout to keep working trees small even when exploring unfamiliar repos. For measuring DX, use the SPACE framework (Satisfaction, Performance, Activity, Collaboration, Efficiency) rather than just lines of code. Discuss git shortlog -sn to see who's contributing across repos, and git log --format='%an' --since=1month for engagement tracking.

Behavioral: Staff+ Leadership (All Companies)

These questions appear in every Staff+ loop. Google L6: "Tell me about the most technically complex project you've led — what was the hardest decision?" Meta L7: "Describe a time you unblocked an entire organization — how did you communicate the path forward?" Amazon L7 (Principal): "Tell me about a time you had to say no to a VP-level request — how did you handle it?" Microsoft L6–L7: "Describe a time you influenced a technical decision across multiple teams without direct authority." Netflix L7: "Tell me about a time you made a decision that was unpopular but correct — what was the outcome?" Apple L7: "Describe a situation where you had to choose between two opposing technical approaches — how did you resolve it with data?" Stripe L6: "Tell me about a time you improved developer productivity by an order of magnitude."

For all answers, use the STAR framework. Staff+ answers must demonstrate: (1) org-wide scope (not team-level), (2) technical depth with trade-offs, (3) influencing without authority, (4) mentorship and team building, and (5) learning from failure.

Staff+ interviews assess four dimensions: technical depth (can you reason about complex systems at scale?), architectural thinking (can you design for trade-offs?), organizational impact (can you drive alignment without authority?), and technical strategy (can you set technical direction?). The Git hosting system design question tests knowledge of Git internals: objects (blob, tree, commit, tag), refs (branches, tags, HEAD), packfiles (thin vs thick, bitmap indexes), and protocol negotiation (want/have, shallow). At GitHub's scale, every request goes through a load balancer to a Git frontend that performs authentication, authorization, and repository routing. The git data plane is separate from the control plane — objects are served from a distributed cache with NFS/EBS as a fallback. Packfile bitmaps speed up clone and fetch by encoding reachability. For monorepos, the key insight is that most engineers only touch a fraction of files — partial clone (--filter=tree:0) defers object fetching, sparse checkout limits the working tree, and git fsmonitor uses filesystem events instead of lstat() calls. CI in monorepos requires dependency graph analysis to determine which projects are affected by a change — tools like Bazel, Nx, or Turborepo compute the affected graph. Actions scale requires horizontal runner autoscaling, job partitioning (matrix), and artifact caching. The coordinator on the Actions team manages the queue, dispatches jobs to runner groups, and handles concurrency. Behavioral questions at the Staff+ level focus on scope — answers should demonstrate org-wide impact, mentoring, and technical leadership without direct authority.

Company-Specific Preparation: For Google, practice distributed systems design — they care about trade-offs, consistency models, and fault tolerance. For Meta, focus on performance optimization at scale — they value concrete numbers and benchmarks. For Amazon, anchor every answer to a Leadership Principle — prepare 3 stories per principle. For Microsoft, understand their actual Git scaling work (VFS, Scalar, Azure Repos) — they want to see that you've studied their engineering blogs. For Netflix, emphasize CD, chaos engineering, and automation. For Apple, focus on cross-team collaboration and dealing with legacy systems. For Stripe, demonstrate API design taste and reliability mindset. For LinkedIn, prepare for code review and merge queue system design — they value throughput and fairness in reviewer assignment. For Airbnb, focus on migration and refactoring at scale — they want to see clean extraction strategies with minimal blast radius. For DoorDash, emphasize dependency management and cross-team coordination — they care about avoiding blocker chains. For Twitter/X, demonstrate open source community expertise — contribution workflows, bot automation, and community health metrics matter most.

Security and supply chain questions at the Staff+ level test your ability to think about the full attack surface: signed commits, secret detection, branch protection, and SBOM generation. The key insight is that Git security is not just about cryptography — it's about process automation (hooks, status checks, policies) and auditability (every action leaves a trace in the reflog). Incident management questions separate Staff from Senior by testing your rollback strategy under pressure. The canonical answer always begins with git revert (not git reset) because preserving history is paramount at scale. Developer Experience (DX) and inner source questions assess organizational thinking — can you design a contribution workflow that thousands of engineers will adopt voluntarily? The answer hinges on reducing friction: fast CI, clear ownership, and low ceremony for the contributor. Inner source succeeds when the path of least resistance is also the correct path.
