Monorepo Management

schedule2 min readsignal_cellular_4_barAdvanced / Staff+
staff+gitmonoreposcale

Scale Git to handle thousands of developers and millions of files using partial clone, sparse checkout, shallow clones, and the same techniques powering Google, Meta, and Microsoft monorepos.

quick_reference Staff+ Cheat Sheetexpand_more

Core Commands

  • git clone --filter=blob:none <url> — partial clone: fetch commits/trees but defer blobs.
  • git sparse-checkout set --cone src/ — limit working tree to specific directories.
  • git maintenance start — background maintenance (fetch, commit-graph, loose-objects).

Staff+ Patterns

  • Combine partial clone + sparse checkout for monorepo performance: --filter=tree:0 defers ALL object fetching, --cone mode limits to your team's directory.
  • Use git fsmonitor--daemon with watchman to eliminate lstat() calls — makes git status instant even in large working trees.

Watch Outs

  • Partial clone requires Git 2.26+ and server-side support (GitHub supports it since 2021). Older Git versions fall back to full clone.
  • Blobless clones (--filter=blob:none) still fetch blobs on demand — the first git log -p will be slow as blobs are lazily fetched.

Interview Angle

Meta L6–L7 asks about monorepo architecture. Key insight: 90% of engineers only touch 10% of the codebase. Partial clone + sparse checkout + fsmonitor make monorepos practical at 10K+ engineer scale. CI must use dependency graph analysis to build only affected projects.

Partial Clone

Partial clone defers blob downloads until needed. Use --filter=blob:none to fetch commits and trees only — blobs download on checkout. --filter=tree:0 skips trees too. --filter=blob:limit=1M fetches small blobs immediately, defers large ones. Pair with git sparse-checkout for maximum efficiency. Promisor remotes fetch missing objects on demand.

language-bash
# Partial clone with blob:none (recommended)
git clone --filter=blob:none <url>
# Initial: ~150MB instead of ~5GB for large monorepos

# Partial clone with tree:0 (deepest partial)
git clone --filter=tree:0 <url>
# Only commit objects initially — trees + blobs on demand

# Blob size limit filter
git clone --filter=blob:limit=1M <url>
# Blobs >1MB are deferred

# Configure promisor remote (optional)
git config remote.origin.promisor true
git config remote.origin.partialclonefilter blob:none

Sparse Checkout

Sparse checkout limits which files appear in your working tree. Cone mode (git sparse-checkout set --cone) is the modern approach: specify top-level directories, and Git only populates those. Parents are included implicitly. This dramatically reduces git status time and disk usage. Combine with partial clone for optimal monorepo performance.

language-bash
# Enable sparse checkout (modern cone mode)
git sparse-checkout init --cone

# Set directories to work on
git sparse-checkout set src/frontend src/shared

# Add additional directories
git sparse-checkout add docs

# Check current sparse checkout state
git sparse-checkout list

# Disable sparse checkout (checkout everything)
git sparse-checkout disable

# Legacy non-cone mode (pattern-based)
git sparse-checkout init  # without --cone
echo "src/frontend/*" >> .git/info/sparse-checkout

Shallow Clones

Shallow clones truncate history with --depth=N. Useful for CI (only need latest commit) and exploratory cloning. --depth=1 fetches one commit — fastest possible clone. --shallow-since= fetches commits since a date. --shallow-exclude= excludes commits reachable from a ref. Unshallow later with git fetch --unshallow.

language-bash
# Shallow clone (single commit — fastest)
git clone --depth=1 <url>

# Shallow clone with history since a date
git clone --shallow-since="2024-01-01" <url>

# Shallow clone excluding specific refs
git clone --shallow-exclude=v1.0 <url>

# Convert shallow to full clone
git fetch --unshallow

# Deepen shallow clone by N commits
git fetch --depth=100

# CI optimization: shallow clone + fetch PR merge commit
# GitHub Actions
git fetch --depth=1 origin ${{ github.event.pull_request.merge_commit_sha }}

filter-repo Migration

When migrating from or to monorepo, git filter-repo is the modern replacement for git filter-branch and BFG Repo-Cleaner. It's 10-100x faster, handles edge cases correctly, and natively supports Python for custom callbacks. Use for splitting monorepos into sub-repos, combining repos, or removing history branches.

language-bash
# Install git-filter-repo
pip install git-filter-repo
# or: brew install git-filter-repo

# Split a monorepo — extract a subdirectory into its own repo
git filter-repo --path src/frontend/ --path src/shared/

# Combine multiple repos into a monorepo
git filter-repo --path-rename old-repo/:new-repo/old-repo/

# Remove all but a specific set of paths
git filter-repo --path-glob '*.py' --path-glob '*.js' --invert-paths

# Remove large files from history
git filter-repo --strip-blobs-bigger-than 10M

# Analyze repository history
git filter-repo --analyze
# Results in .git/filter-repo/analysis/

CI/CD for Monorepos

Monorepo CI challenges: only build what changed. Use git diff to detect changed paths, dependency graph analysis (Bazel, Nx, Turborepo) to determine affected projects. GitHub Actions offers path filters. For large monorepos, implement build graph caching and distributed builds. Use merge queues to batch compatible changes.

language-bash
# GitHub Actions: Path-based triggers
on:
  pull_request:
    paths:
      - 'src/frontend/**'
      - 'src/shared/**'

# Detect changed projects in monorepo
CHANGED=$(git diff --name-only HEAD~1)
if echo "$CHANGED" | grep -q "^src/frontend/"; then
  cd src/frontend && npm run build
fi

# Bazel query for affected targets
bazel query 'kind(".*_binary", rdeps(//..., set($(git diff --name-only HEAD~1))))'

# Nx affected command
npx nx affected:test --base=main --head=HEAD

# Monorepo CI flow
# 1. Filter: identify changed projects
# 2. Build: only changed + dependents
# 3. Test: run affected test suites
# 4. Lint: scope to changed files
# 5. Package: publish only changed packages
lightbulbDeep Dive & Key Takeawaysexpand_more

Monorepo management at FAANG scale relies on three pillars: client-side filtering (partial clone + sparse checkout), server-side optimization (reachability bitmaps, MIDX, commit-graph), and build system integration (Bazel, Buck, Please). Google's Piper monorepo (2B+ lines, 25K+ developers) uses a proprietary VCS but Git monorepos can approach similar scale with Scalar (Microsoft's Git extension). Key benchmarks: partial clone reduces initial clone from 4.5 GB to 145 MB (97% reduction); sparse checkout reduces git status from 45s to 0.3s; commit-graph speeds up merge-base from 12s to 0.8s. The critical insight: Git is not slow for monorepos — it's slow when fetching/checking things you don't need.
