Trunk-based vs Multi-Repo

schedule3 min readsignal_cellular_4_barAdvanced / Staff+
staff+gitarchitecturefaang

Compare monorepo and multi-repo architectures used by the largest tech companies — Google Piper, Meta Mononoke, and Microsoft VFS for Git — with migration strategies between approaches.

quick_reference Staff+ Cheat Sheetexpand_more

Core Concepts

  • Trunk-based: one repo, short-lived feature branches, merge often. Simpler coordination, atomic cross-service changes. Needs strong CI.
  • Multi-repo: separate repos per service/team. Team autonomy, independent deployment. Needs dependency versioning discipline.
  • Monorepo: one repo, all code. Atomic commits, unified tooling. Needs performance tooling (partial clone, sparse checkout).

Staff+ Patterns

  • Start with a monorepo, extract to multi-repo when team boundaries stabilize. Extraction is easier than consolidation: git filter-repo --path service/.
  • In a multi-repo setup, use git submodule or tools/workspace only for shared build configs. Never for active development — the overhead is too high.

Watch Outs

  • Monorepo tooling investment is front-loaded; multi-repo costs appear later (dependency hell, version skew). Choose based on team maturity.
  • Multi-repo doesn't mean "no monorepo tools" — you still need cross-repo search, CI, and dependency tracking.

Interview Angle

Uber L6 and DoorDash L6 ask about repo strategy. The Staff+ answer is: "It depends on org maturity and tooling investment." Discuss trade-offs rather than advocating one true way. Know that Google/Meta use monorepos, Amazon/Uber use multi-repo, and both work at scale.

Trunk-Based Development

Trunk-based development: all developers merge to a single branch (trunk) multiple times daily. Short-lived feature branches (< 48 hours) behind feature flags. Benefits: no merge hell, fast CI feedback, easy releases. Prerequisites: robust feature flags, comprehensive test suite, automated CI/CD, and team discipline. Paul Hammant's trunk-based development defines 'short-lived' as < 2 days, with branches of at most a few commits.

language-text
# Trunk-based development workflow
git checkout main
git pull --rebase origin main
git checkout -b feat/user-profile-ff
# Make small changes, commit often
git commit -m "feat: add user profile page behind flag"
git push origin feat/user-profile-ff
# Open PR, get review, squash-merge to main

# Feature flags in code
if (FeatureFlags.isEnabled("user-profile-v2")) {
    renderNewProfile()
} else {
    renderLegacyProfile()
}

# Release from trunk
git tag v2.3.0 main
git push origin v2.3.0
# CI builds and deploys the tagged commit

Multi-Repo Architecture

Multi-repo (polyrepo): each project, service, or library has its own Git repository. Benefits: independent versioning, smaller clone sizes, isolated CI/CD, team autonomy. Drawbacks: cross-repo refactoring is painful, dependency management complexity (pinning versions), duplication of tooling and config, inconsistent coding standards. Common in microservice architectures where each service is independently deployable.

language-text
# Multi-repo dependency management challenges

# Each repo has its own version
git clone https://github.com/org/service-a.git
git clone https://github.com/org/service-b.git
git clone https://github.com/org/shared-lib.git

# Cross-repo changes require coordinated releases
# 1. Update shared-lib, publish v2.1.0
# 2. Update service-a to use shared-lib@v2.1.0
# 3. Update service-b to use shared-lib@v2.1.0
# 4. Deploy service-a, then service-b

# Workaround: git submodules (not recommended)
git submodule add https://github.com/org/shared-lib.git libs/shared
# Or: package manager references (npm, pip, go modules)
# package.json
{ "dependencies": { "shared-lib": "^2.1.0" } }

Google Piper Architecture

Piper is Google's proprietary version control system, replacing Perforce. Key features: Single trunk — no branching culture; every change lands on main. Transactional commits — atomic across the entire tree. Presubmits — every CL runs full test suite before landing. CitC (Clients in the Cloud) — cloud-based workspace, no local clones. Global namespace — one tree visible to all 25K+ developers. Piper stores metadata in Bigtable, content in Colossus.

language-text
# Piper workflow (conceptual — not Git)
# But you can approximate with:

# 1. Create changelist (CL)
# - Edit files locally (CitC workspace)
# - View diff: g4 diff

# 2. Run presubmit checks
# - All affected BUILD targets tested
# - g4 presubmit

# 3. Request review
# - Critique: Google's code review tool
# - +2 from owner = ship it

# 4. Submit to trunk
# - Transactional: all or nothing
# - Automatically merges with concurrent changes
# - Can auto-revert on rollback

# Git approximation: squash-merge with linear history
git pull --rebase main
git push origin HEAD:main  # requires fast-forward

Meta Mononoke Architecture

Mononoke is Meta's Git-compatible server, replacing Mercurial (which replaced Git internally). Key features: Pushrebase — on push, rebase commits onto main tip for linear history. Warm bookmarks cache — low-latency ref lookups. Segmented changelog — efficient history fetching for large repos. Queries — custom SQL-like query interface for Git data. LFS (Large File Storage) — integrated with Mononoke backend. Serves millions of commits per day for Meta's monorepo.

language-text
# Mononoke's Pushrebase (conceptual)
# Git-simulated pushrebase:

# Instead of:
git push origin feature
# (may create merge commit on server)

# Pushrebase workflow:
git fetch origin main
git rebase origin/main
git push origin feature:main --force-with-lease
# Server rebases onto latest main before accepting

# Mononoke server config (simplified)
# bookmarks:
#   - name: main
#     pushrebase: true
#     hooks:
#       - phabricator_acceptance
#       - ci_passed

# Git LFS in Mononoke
git lfs track "*.psd"
git lfs track "*.zip"
git add .gitattributes
git commit -m "config: add LFS tracking for binaries"

Migration Strategies

Migrating between architectures is high-risk: Multi-repo → Monorepo: preserve history per repo via git filter-repo --path-rename, combine with scripted merge, build dependency graph. Monorepo → Multi-repo: split with git filter-repo --path, establish package management, coordinate releases. SVN → Git: use git svn clone with --stdlayout for full history. Mercurial → Git: hg-fast-export or git-cinnabar. Critical: freeze writes, migrate, verify, switch read-only to new system.

language-text
# Multi-repo to monorepo migration
# 1. Collect all repos
repos=(service-a service-b shared-lib)

# 2. Rewrite each with directory prefix
git clone service-a && cd service-a
git filter-repo --path-rename :service-a/
cd ..

# 3. Create monorepo and merge
mkdir monorepo && cd monorepo
git init
for repo in "${repos[@]}"; do
  git remote add $repo ../$repo
  git fetch $repo
  git merge --allow-unrelated-histories $repo/main --no-edit
done

# Monorepo to multi-repo
# Split with filter-repo
git clone monorepo && cd monorepo
git filter-repo --path service-a/ --refs main

# SVN to Git migration
git svn clone https://svn.example.com/repo \
  --stdlayout --prefix svn/ \
  --authors-file=authors.txt \
  my-git-repo
lightbulbDeep Dive & Key Takeawaysexpand_more

The monorepo vs multi-repo debate is ultimately about tradeoffs. Monorepos excel at: atomic cross-project refactoring, consistent tooling, unified versioning, simplified dependency management, and discoverability. Multi-repos excel at: team autonomy, independent deployability, smaller surface area, and reduced cognitive load. The trend among FAANG is convergence: Google uses a monorepo (Piper), Meta migrated to a monorepo (Mononoke), Microsoft uses a monorepo (Windows + VFS for Git). But these companies built custom infrastructure to overcome Git's limitations at scale. For most organizations, a well-configured Git monorepo (partial clone + sparse checkout + commit-graph + bitmaps) serves up to 500 developers effectively. Beyond that, consider Scalar for Git or evaluate custom solutions. The worst choice is neither — it's making a choice based on hype rather than evidence. Measure your own pain points: cross-project refactoring frequency, CI cycle time, onboarding friction, and dependency management overhead.
